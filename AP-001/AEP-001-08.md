# 1. Purpose

This document defines the repository strategy for 86-vibe.

The objective is to separate architectural assets, implementation assets and validation assets into independent repositories with clearly defined responsibilities.

This strategy improves maintainability, simplifies versioning and enables independent evolution of platform components.

---

# 2. Repository Strategy

86-vibe is organised into multiple repositories rather than a single monolithic repository.

Each repository has one primary responsibility.

Repositories SHALL communicate through documented interfaces and shared versioned releases rather than direct dependency upon unpublished changes.

Each repository SHALL remain independently buildable and independently versioned.

---

# 3. Repository Principles

The repository strategy follows these principles.

### RP-001 Single Responsibility

Each repository SHALL have one primary purpose.

---

### RP-002 Independence

Repositories SHALL be capable of independent development.

---

### RP-003 Versioned Interfaces

Repositories SHALL consume released versions of shared assets wherever practical.

---

### RP-004 Documentation First

Every repository SHALL include complete documentation.

---

### RP-005 Shared Standards

All repositories SHALL comply with the engineering standards defined by the platform repository.

---

# 4. Repository Overview

Version 1.0 consists of three repositories.

---

## Repository 1

### Name

86-vibe-platform

### Purpose

Defines the engineering platform.

Responsibilities

- Architecture
- Standards
- Templates
- Documentation
- AI Agent Definitions
- Engineering Workflows
- Governance
- ADRs

The platform repository is the authoritative source for engineering knowledge.

No implementation-specific code SHALL define platform architecture outside this repository.

---

## Repository 2

### Name

86-vibe-cli

### Purpose

Provides executable tooling for the platform.

Responsibilities

- Bootstrap Engine
- Project Creation
- Validation
- Automation
- Template Processing
- Command Line Interface

The CLI repository SHALL implement the standards defined by the platform repository.

---

## Repository 3

### Name

86-vibe-example

### Purpose

Provides a reference implementation.

Responsibilities

- Demonstrate platform capabilities
- Validate platform releases
- Provide implementation examples
- Support regression testing

The example repository SHALL always represent current best practice.

# 6. Repository Dependencies

Repository dependencies SHALL follow the hierarchy below.

```text
86-vibe-platform
        │
        ├──────────────┐
        ▼              ▼
86-vibe-cli     86-vibe-example
        │
        ▼
86-vibe-example
```

The dependency rules are:

- Platform SHALL NOT depend upon CLI.
- Platform SHALL NOT depend upon Example.
- CLI MAY depend upon Platform.
- Example MAY depend upon Platform.
- Example MAY depend upon CLI.

Circular dependencies SHALL NOT be introduced.

---

# 7. Repository Ownership

Each repository SHALL have a defined owner.

| Repository | Owner |
|------------|-------|
| 86-vibe-platform | Product Owner |
| 86-vibe-cli | Product Owner |
| 86-vibe-example | Product Owner |

Future releases MAY delegate repository maintenance while retaining Product Owner governance.

---

# 8. Versioning Strategy

Each repository SHALL maintain its own semantic version.

Version numbers SHALL evolve independently.

Example:

| Repository | Version |
|------------|----------|
| 86-vibe-platform | 1.2.0 |
| 86-vibe-cli | 1.1.4 |
| 86-vibe-example | 1.2.0 |

Major releases SHOULD remain aligned where practical.

Repository releases SHALL identify compatible versions of dependent repositories.

---

# 9. Branch Strategy

The following branches SHALL be maintained.

## main

Production-ready code and documentation.

---

## develop

Integration branch for completed work awaiting release.

---

## feature/*

Implementation of individual features.

Examples

feature/bootstrap-engine

feature/openrouter-routing

feature/jira-agent

---

## release/*

Preparation for a formal release.

---

## hotfix/*

Urgent production fixes.

---

Every Pull Request SHALL target the appropriate branch.

Direct commits to the **main** branch SHOULD be avoided except in exceptional circumstances approved by the Product Owner.

---

# 10. Repository Standards

Every repository SHALL contain, at minimum:

- README.md
- LICENSE
- CHANGELOG.md
- CONTRIBUTING.md
- CODE_OF_CONDUCT.md
- .gitignore

Repositories SHOULD also include:

- Architecture references
- Issue templates
- Pull Request templates
- Development documentation

Repository documentation SHALL reference the authoritative architecture maintained within **86-vibe-platform**.

---

# 11. Summary

The repository strategy provides a clear separation between platform definition, platform implementation and platform validation.

This separation enables the repositories to evolve independently while maintaining architectural consistency.

By assigning a single primary responsibility to each repository, 86-vibe reduces coupling, improves maintainability and simplifies future expansion.

The repository strategy defined within this document SHALL be followed by all future development of the platform.
