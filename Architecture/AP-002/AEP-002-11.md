# 1. Purpose

This document defines the Document Validation Framework (DVF) for the 86-vibe platform.

The Document Validation Framework provides a deterministic mechanism for discovering, validating, classifying, and reporting on all governed documentation within the platform. It ensures that architecture documents, specifications, prompts, and supporting documentation comply with platform standards before implementation activities begin.

The framework is a foundational governance capability supporting the "Documentation First" engineering principle established in AP-001.

---

# 2. Scope

This specification defines:

- Document discovery
- Document classification
- Metadata validation
- Naming validation
- Structural validation
- Cross-reference validation
- Dependency validation
- Validation reporting
- CLI integration
- Testing requirements

This document does not define document authoring standards or Markdown formatting conventions beyond those required for validation.

---

# 3. Design Principles

The Document Validation Framework SHALL adhere to the following principles.

## 3.1 Documentation First

Documentation SHALL be validated before implementation activities begin.

## 3.2 Deterministic

The same document set SHALL always produce the same validation results.

## 3.3 Non-Destructive

Validation SHALL never modify repository contents.

## 3.4 Extensible

Additional validation rules SHALL be added without breaking existing documents.

## 3.5 Repository Independent

Validation SHALL operate consistently across all supported repository types.

---

# 4. Architectural Overview

The Document Validation Framework SHALL operate as an independent platform service.

```text
                  CLI
                   │
                   ▼
      Document Validation Framework
                   │
     ┌─────────────┼──────────────┐
     │             │              │
     ▼             ▼              ▼
 Discovery     Validation      Reporting
                   │
                   ▼
          Validation Results
```

Other platform components SHALL consume validation results through this framework.

---

# 5. Responsibilities

The Document Validation Framework SHALL be responsible for:

- Document discovery
- Metadata extraction
- Naming validation
- Structural validation
- Cross-reference validation
- Dependency validation
- Validation reporting
- Result classification

The framework SHALL NOT:

- Modify documents
- Automatically correct validation failures
- Approve architecture documents
- Generate documentation

---

# 6. Document Discovery

The framework SHALL automatically discover governed documents within supported repositories.

Supported document categories SHALL include:

| Category | Description |
|----------|-------------|
| Architecture | Architecture packages and ADRs |
| Specifications | Technical specifications |
| Prompts | Prompt templates |
| Guides | User and developer documentation |
| Policies | Governance documentation |
| Reference | Supporting reference material |

Discovery SHALL be recursive and deterministic.

---

# 7. Document Identification

Each governed document SHALL possess a unique identifier.

Examples include:

```text
AEP-001-07
AEP-002-04
ADR-0003
```

Identifiers SHALL be unique within the platform.

Duplicate identifiers SHALL produce validation failures.

---

# 8. Naming Convention Validation

Document filenames SHALL conform to approved naming conventions.

Examples:

```text
AEP-002-11-document-validation-framework.md

ADR-0003-ai-provider-selection.md
```

Validation SHALL verify:

- Prefix
- Numeric sequence
- Descriptive suffix
- File extension

---

# 9. Metadata Validation

The framework SHALL validate required document metadata.

Minimum metadata SHALL include:

- Architecture Package
- Document Identifier
- Title
- Platform
- Status
- Owner

Missing required metadata SHALL constitute a validation failure.

---

# 10. Structural Validation

The framework SHALL validate document structure.

Minimum structure SHALL include:

- Title
- Document Control
- Numbered sections
- Approval section

Future document types MAY introduce additional structural requirements.

---

# 11. Cross-Reference Validation

The framework SHALL validate references between governed documents.

Validation SHALL verify:

- Referenced document exists
- Referenced identifier is valid
- Duplicate references are permitted
- Invalid references are reported

Missing referenced documents SHALL generate validation failures.

---

# 12. Dependency Validation

Documents MAY declare dependencies.

Example:

```text
Dependencies

- AEP-001-07
- AEP-002-05
```

Validation SHALL verify:

- Dependency existence
- Dependency identifier validity
- Duplicate dependencies

Dependency cycles SHALL generate warnings.

---

# 13. Validation Rules

The framework SHALL support rule-based validation.

Minimum rule categories SHALL include:

| Rule Category | Purpose |
|---------------|---------|
| Naming | Filename validation |
| Metadata | Required metadata |
| Structure | Document organization |
| References | Cross-document references |
| Dependencies | Dependency graph |
| Repository | Repository placement |

Future rule categories MAY be added.

---

# 14. Validation Severity

Validation results SHALL use standardized severity levels.

| Severity | Description |
|----------|-------------|
| Information | Informational only |
| Warning | Valid but requires attention |
| Error | Validation failure |
| Critical | Validation cannot continue |

Severity definitions SHALL remain consistent across the platform.

---

# 15. Validation Lifecycle

Validation SHALL follow the sequence below.

```text
Discover Documents
         │
         ▼
Extract Metadata
         │
         ▼
Run Validation Rules
         │
         ▼
Validate References
         │
         ▼
Validate Dependencies
         │
         ▼
Generate Results
```

---

# 16. CLI Integration

The following CLI commands SHALL integrate with the Document Validation Framework.

```text
86vibe arch validate

86vibe repo docs

86vibe doctor
```

Future commands MAY include:

```text
86vibe docs validate

86vibe docs graph

86vibe docs inventory
```

---

# 17. Reporting

Validation reports SHALL include:

- Documents discovered
- Documents validated
- Validation duration
- Warnings
- Errors
- Critical failures
- Summary statistics

Reports SHALL support both human-readable and machine-readable formats.

---

# 18. Logging Requirements

The Logging Service SHALL record:

- Validation execution
- Documents discovered
- Validation duration
- Validation failures
- Rule execution
- Summary results

Document contents SHALL NOT be logged.

---

# 19. Performance Requirements

The framework SHALL:

- Cache document discovery where practical
- Avoid repeated parsing
- Support incremental validation
- Scale to large documentation repositories

Validation SHALL remain deterministic regardless of repository size.

---

# 20. Extensibility

Future validation capabilities MAY include:

- Mermaid diagram validation
- Markdown linting
- Link validation
- Table validation
- Spell checking
- Diagram dependency analysis
- Prompt validation integration
- Documentation quality scoring

These capabilities SHALL be introduced without changing the framework architecture.

---

# 21. Testing Requirements

Automated tests SHALL verify:

- Document discovery
- Metadata extraction
- Filename validation
- Metadata validation
- Structural validation
- Cross-reference validation
- Dependency validation
- Duplicate identifier detection
- Severity classification
- Report generation

Representative document sets SHALL be used during testing.

---

# 22. Definition of Done

The Document Validation Framework SHALL be considered complete when:

- Document discovery functions correctly
- Metadata validation succeeds
- Structural validation succeeds
- Cross-reference validation succeeds
- Dependency validation succeeds
- CLI integration is complete
- Reports are generated correctly
- Automated tests pass
- Documentation is complete
