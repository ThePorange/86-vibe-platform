---

# 1. Purpose

This document defines the standard Local Development Workflow for the 86-vibe platform.

The workflow establishes a consistent engineering process for all contributors, whether development is performed manually or through AI-assisted tools such as Cursor and Claude Code.

The objective is to ensure every feature is implemented through the same governed lifecycle, maintaining consistency, quality, traceability, and repeatability across the platform.

This workflow implements the engineering principles established in AP-001 and utilizes the platform services defined throughout AP-002.

---

# 2. Scope

This specification defines:

- Local developer workflow
- Repository initialization
- Development lifecycle
- Architecture validation
- AI-assisted implementation workflow
- Testing workflow
- Documentation updates
- Build validation
- Git integration
- Completion criteria

This document does not define release workflows or production deployment processes.

---

# 3. Design Principles

The Local Development Workflow SHALL adhere to the following principles.

## 3.1 Architecture First

Implementation SHALL begin only after architecture approval.

## 3.2 Documentation First

Documentation SHALL be updated before implementation is considered complete.

## 3.3 Git First

All work SHALL occur within a Git repository.

## 3.4 Validation Before Completion

Validation SHALL precede every implementation completion.

## 3.5 AI Assisted, Human Governed

AI tools SHALL assist implementation while human engineers retain approval authority.

---

# 4. Workflow Overview

Every implementation SHALL follow the standard engineering workflow.

```text
Clone Repository
        │
        ▼
Initialize Environment
        │
        ▼
Validate Repository
        │
        ▼
Review Architecture
        │
        ▼
Implement Feature
        │
        ▼
Execute Validation
        │
        ▼
Execute Tests
        │
        ▼
Review Changes
        │
        ▼
Commit to Git
```

No workflow stage SHALL be skipped without explicit approval.

---

# 5. Repository Preparation

Developers SHALL begin by obtaining the repository.

Typical workflow:

```bash
git clone ...

cd repository

86vibe doctor
```

The repository SHALL successfully pass initial validation before implementation begins.

---

# 6. Environment Initialization

The local environment SHALL include:

- Supported Python version
- Required dependencies
- Platform configuration
- AI provider configuration
- MCP configuration where applicable

Initialization MAY occur locally or within the Docker Development Environment defined in AEP-002-14.

---

# 7. Repository Validation

Prior to implementation, developers SHALL execute repository validation.

Minimum validation SHALL include:

- Repository Services validation
- Configuration validation
- Architecture validation
- Prompt validation
- Documentation validation

Validation failures SHALL be resolved before implementation begins.

---

# 8. Architecture Review

Implementation SHALL be based upon approved architecture.

Developers SHALL verify:

- Required architecture documents exist
- Architecture status is Approved
- Dependencies have been satisfied
- No superseding ADR exists

Architecture SHALL remain the authoritative implementation specification.

---

# 9. Feature Implementation

Implementation SHALL:

- Follow approved architecture
- Conform to repository standards
- Reuse existing platform services
- Maintain coding standards
- Avoid unnecessary architectural changes

Implementation SHALL NOT modify approved architecture unless an ADR has been approved.

---

# 10. AI-Assisted Development

When AI tools are used:

- Approved architecture SHALL provide implementation guidance.
- Prompt templates SHALL originate from the Prompt Management Framework.
- AI-generated code SHALL be reviewed by a human.
- AI-generated code SHALL satisfy all testing requirements.

AI assistance SHALL not bypass governance.

---

# 11. Documentation Updates

Documentation SHALL be updated whenever implementation changes require it.

Documentation MAY include:

- User documentation
- Developer documentation
- API documentation
- Architecture references
- Examples

Documentation updates SHALL be committed together with implementation changes whenever practical.

---

# 12. Validation Workflow

Following implementation, developers SHALL execute platform validation.

Validation SHALL include:

```text
Repository Validation
        │
        ▼
Configuration Validation
        │
        ▼
Architecture Validation
        │
        ▼
Prompt Validation
        │
        ▼
Documentation Validation
```

All required validation SHALL succeed.

---

# 13. Testing Workflow

Developers SHALL execute automated testing.

Minimum testing SHALL include:

- Unit tests
- Integration tests
- End-to-End tests where applicable

Testing SHALL follow AEP-002-12.

Implementation SHALL not be considered complete if required tests fail.

---

# 14. Build Validation

Where applicable, developers SHALL execute the build process.

Build validation SHALL verify:

- Successful package generation
- Build integrity
- Artifact validation
- Metadata generation

Build failures SHALL be resolved prior to completion.

---

# 15. Git Workflow

Following successful validation:

- Review implementation
- Review documentation
- Review tests
- Commit changes

The Local Development Workflow SHALL not automate Git commits.

Commit messages SHALL follow the platform contribution guidelines defined in future architecture packages.

---

# 16. Logging Requirements

The Logging Service SHALL record:

- Validation execution
- Test execution
- Build execution
- CLI activity
- Workflow failures

Normal development activity SHALL remain traceable through platform logs.

---

# 17. Error Recovery

If any workflow stage fails:

- Execution SHALL stop.
- Errors SHALL be reported.
- Corrective action SHALL be taken.
- Validation SHALL be repeated before continuing.

Developers SHALL not continue beyond unresolved validation failures.

---

# 18. Performance Requirements

The Local Development Workflow SHALL:

- Minimize unnecessary repetition
- Support incremental development
- Support rapid validation
- Avoid redundant execution where practical

Performance optimization SHALL not reduce validation quality.

---

# 19. Supported Development Modes

The platform SHALL support:

| Mode | Description |
|------|-------------|
| Local Native | Development directly on the host operating system |
| Docker | Development within the Docker Development Environment |
| AI Assisted | Cursor, Claude Code, or equivalent AI tooling |
| Offline | Development without AI provider connectivity where practical |

All modes SHALL follow the same workflow.

---

# 20. Developer Responsibilities

Developers SHALL:

- Follow approved architecture
- Execute validation
- Execute tests
- Review AI-generated code
- Maintain documentation
- Preserve repository quality

Human approval SHALL remain mandatory before architecture baselines are modified.

---

# 21. CLI Integration

The Local Development Workflow SHALL utilize existing CLI capabilities including:

```text
86vibe doctor

86vibe init

86vibe repo validate

86vibe arch validate

86vibe config validate

86vibe prompt validate
```

Future workflow commands MAY be introduced through subsequent architecture packages.

---

# 22. Workflow Lifecycle

The complete development lifecycle SHALL follow the sequence below.

```text
Repository Ready
        │
        ▼
Validate Environment
        │
        ▼
Review Architecture
        │
        ▼
Implement
        │
        ▼
Validate
        │
        ▼
Test
        │
        ▼
Build
        │
        ▼
Review
        │
        ▼
Commit
```

This lifecycle SHALL represent the standard engineering process for all repositories.

---

# 23. Definition of Done

An implementation SHALL be considered complete when:

- Approved architecture has been implemented
- Documentation has been updated
- Repository validation succeeds
- Configuration validation succeeds
- Documentation validation succeeds
- Automated tests pass
- Build validation succeeds where applicable
- Human review has been completed
- Changes are committed through the normal Git workflow
