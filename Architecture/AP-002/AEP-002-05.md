# 1. Purpose

This document defines the architecture, structure, lifecycle, and governance of configuration used throughout the 86-vibe platform.

Configuration SHALL be:

- Deterministic
- Portable
- Version controlled where appropriate
- Secure
- Human-readable
- Machine-validated

This document establishes the single authoritative configuration model used by all 86-vibe repositories.

---

# 2. Scope

This specification defines:

- Configuration hierarchy
- Configuration file formats
- Configuration precedence
- Environment variable usage
- Secret management
- Configuration validation
- Default values
- Configuration discovery
- Runtime configuration loading

This document does not define provider-specific configuration schemas, which SHALL be covered in later architecture documents.

---

# 3. Design Principles

Configuration SHALL satisfy the following principles.

## 3.1 Human Readable

Configuration SHALL be understandable without specialized tooling.

## 3.2 Version Controlled

Project configuration SHALL be committed to Git unless it contains secrets.

## 3.3 Environment Independent

Repositories SHALL be portable between developer workstations.

## 3.4 Secure by Default

Secrets SHALL never be committed to source control.

## 3.5 Deterministic

Given the same configuration inputs, the platform SHALL behave identically.

## 3.6 Extensible

Future providers and services SHALL be added without breaking existing configuration files.

---

# 4. Configuration Hierarchy

Configuration SHALL exist at four logical levels.

```text
CLI Arguments
        │
Environment Variables
        │
Project Configuration
        │
Platform Defaults
```

Higher levels SHALL override lower levels.

---

# 5. Configuration Precedence

The platform SHALL resolve configuration using the following order of precedence.

| Priority | Source |
|----------|--------|
| 1 | CLI command arguments |
| 2 | Environment variables |
| 3 | Project configuration |
| 4 | Built-in platform defaults |

No additional precedence rules SHALL exist unless approved through an Architecture Decision Record (ADR).

---

# 6. Project Configuration Location

Every 86-vibe repository SHALL contain a project configuration directory.

```text
.86vibe/
    config.yaml
```

This file SHALL be the authoritative configuration source for the repository.

---

# 7. Configuration Directory Layout

The CLI SHALL create and maintain the following directory structure.

```text
.86vibe/
│
├── config.yaml
├── cache/
├── logs/
├── state/
└── templates/
```

Directory purposes:

| Directory | Purpose |
|-----------|---------|
| `config.yaml` | Project configuration |
| `cache/` | Temporary cached data |
| `logs/` | CLI and platform logs |
| `state/` | Runtime state information |
| `templates/` | Generated templates and cached prompt artifacts |

Additional directories MAY be introduced in future architecture packages.

---

# 8. Configuration File Format

The project configuration SHALL use YAML.

Example:

```yaml
platform:
  name: 86-vibe

repository:
  type: platform
  authoritative: true

ai:
  provider: openrouter
  default_model: anthropic/claude-opus-4

mcp:
  enabled: true

logging:
  level: INFO

governance:
  require_human_approval: true
  architecture_lock: true
```

Configuration SHALL conform to the schema defined by the platform.

# 10. Environment Variables

Environment variables SHALL override values defined in `config.yaml`.

Supported variables SHALL use the `VIBE_` prefix where applicable.

Examples:

```text
VIBE_AI_PROVIDER
VIBE_AI_MODEL
VIBE_LOG_LEVEL
VIBE_MCP_ENABLED
```

Provider credentials SHALL use their provider-defined environment variable names.

Examples:

```text
OPENROUTER_API_KEY
ANTHROPIC_API_KEY
OPENAI_API_KEY
```

Only documented environment variables SHALL be supported.

---

# 11. Secret Management

Secrets SHALL NOT be stored in:

- `config.yaml`
- Git repositories
- Prompt files
- Architecture documents
- Log files

Secrets SHALL be supplied through environment variables or secure operating system credential stores.

The platform SHALL detect missing required credentials before attempting provider connections.

---

# 12. Configuration Validation

Configuration SHALL be validated during:

- `86vibe init`
- `86vibe doctor`
- `86vibe config validate`
- CLI startup
- Runtime initialization

Validation SHALL include:

- YAML syntax validation
- Schema validation
- Required field validation
- Type checking
- Enumeration validation
- File existence
- Directory existence where applicable

Invalid configuration SHALL prevent execution.

---

# 13. Configuration Loading Sequence

Configuration SHALL be loaded using the following sequence.

```text
Load Platform Defaults
          │
          ▼
Load Project Configuration
          │
          ▼
Load Environment Variables
          │
          ▼
Apply CLI Overrides
          │
          ▼
Validate Configuration
          │
          ▼
Freeze Runtime Configuration
```

Once validation succeeds, the runtime configuration SHALL become immutable.

---

# 14. Runtime Configuration Access

Application code SHALL access configuration exclusively through the Configuration Service.

Direct reading of `config.yaml` outside the Configuration Service SHALL NOT be permitted.

This ensures:

- Consistent precedence handling
- Validation guarantees
- Centralized caching
- Future extensibility

---

# 15. Configuration Service Responsibilities

The Configuration Service SHALL provide functionality equivalent to:

- Load configuration
- Validate configuration
- Retrieve configuration values
- Test for configuration existence
- Export configuration as structured data
- Reload configuration where supported

The implementation language and API design are intentionally left to the implementation team.

---

# 16. Default Values

The platform SHALL provide sensible defaults wherever practical.

Examples include:

| Setting | Default |
|---------|---------|
| Logging level | INFO |
| Architecture lock | Enabled |
| Human approval | Required |
| Local cache directory | `.86vibe/cache` |
| Local log directory | `.86vibe/logs` |
| Colored terminal output | Enabled |

Defaults SHALL be documented and deterministic.

---

# 17. Invalid Configuration Handling

Invalid configuration SHALL prevent command execution.

Example output:

```text
CONFIGURATION ERROR

Missing required key:

ai.provider

Run:

86vibe config validate
```

Error messages SHALL:

- Clearly identify the problem
- Identify the configuration location
- Suggest corrective action where possible

---

# 18. Logging Requirements

Configuration loading SHALL record:

- Configuration source
- Validation success
- Validation warnings
- Applied overrides
- Errors

Sensitive values SHALL always be redacted.

Examples of data that SHALL NOT be logged include:

- API keys
- Access tokens
- Passwords
- Secrets
- Private certificates

---

# 19. Future Extension

The configuration schema SHALL support backward-compatible extension.

Future configuration sections SHALL be optional unless introduced through an approved Architecture Decision Record (ADR).

Configuration consumers SHALL ignore unknown sections unless explicitly configured to reject them.

---

# 20. Testing Requirements

Automated tests SHALL verify:

- Platform defaults
- YAML parsing
- Environment variable overrides
- CLI argument overrides
- Configuration precedence
- Missing configuration files
- Invalid schemas
- Unknown properties
- Secret redaction
- Immutable runtime configuration

Tests SHALL be executable using:

```bash
pytest
```

---

# 21. Definition of Done

The Configuration Management subsystem SHALL be considered complete when:

- Configuration loads deterministically
- Precedence rules are enforced
- Validation succeeds for valid configurations
- Invalid configurations are rejected
- Secrets remain external to project configuration
- Runtime configuration is immutable after validation
- Automated tests pass
- Documentation is updated
