# 1. Purpose

This document defines the testing strategy for the 86-vibe platform.

The Testing Strategy establishes the architecture, governance, and implementation standards for validating all platform components. It ensures that every implementation produced by human developers or AI assistants is verified through a consistent, repeatable, and automated testing process before being considered complete.

Testing is a mandatory quality gate supporting the "Architecture Before Implementation", "Documentation First", and "Human Approval" principles established in AP-001.

---

# 2. Scope

This specification defines:

- Testing architecture
- Test categories
- Test organization
- Test execution
- Test data
- Test coverage
- Mocking strategy
- AI implementation testing
- Continuous validation
- Reporting
- Testing governance

This document does not define CI/CD pipeline implementation or deployment testing.

---

# 3. Design Principles

The testing architecture SHALL adhere to the following principles.

## 3.1 Automated First

All repeatable tests SHALL be automated.

## 3.2 Deterministic

Tests SHALL produce consistent results when executed against identical inputs.

## 3.3 Fast Feedback

Developers SHALL be able to execute the complete local test suite before committing changes.

## 3.4 Isolated

Tests SHALL execute independently whenever practical.

## 3.5 Repository Independent

The testing architecture SHALL be consistent across all platform repositories.

---

# 4. Testing Architecture

Testing SHALL be organized into independent layers.

```text
                Test Runner
                     │
     ┌───────────────┼────────────────┐
     │               │                │
     ▼               ▼                ▼
 Unit Tests   Integration Tests   End-to-End Tests
     │               │                │
     └───────────────┼────────────────┘
                     ▼
             Validation Reports
```

Each testing layer SHALL provide increasing confidence while maintaining clear separation of responsibilities.

---

# 5. Test Categories

The platform SHALL support the following test categories.

| Category | Purpose |
|----------|---------|
| Unit | Validate individual components |
| Integration | Validate interaction between components |
| End-to-End | Validate complete workflows |
| Architecture | Validate architecture artifacts |
| Documentation | Validate governed documentation |
| Configuration | Validate configuration loading and precedence |
| Performance | Validate acceptable execution characteristics |

Additional categories MAY be introduced in future architecture packages.

---

# 6. Repository Test Structure

Each repository SHALL contain a dedicated test hierarchy.

```text
tests/
│
├── unit/
├── integration/
├── e2e/
├── fixtures/
├── mocks/
└── data/
```

Repository-specific test directories MAY be added when required.

---

# 7. Unit Testing

Unit tests SHALL validate individual classes, functions, and services.

Requirements:

- No external network access
- No dependency on AI providers
- No dependency on live MCP servers
- Deterministic execution
- Fast execution

Unit tests SHALL use mocks wherever external dependencies exist.

---

# 8. Integration Testing

Integration tests SHALL validate interaction between platform subsystems.

Examples include:

- CLI and Configuration Service
- CLI and Repository Services
- Prompt Framework and AI Provider Layer
- MCP Integration and Logging Service

Integration tests MAY use mock implementations of external services.

---

# 9. End-to-End Testing

End-to-End (E2E) tests SHALL validate complete engineering workflows.

Examples include:

- Repository initialization
- Architecture validation
- Prompt rendering
- AI workflow planning
- Repository diagnostics

End-to-End tests SHALL exercise the platform through its public CLI interface wherever practical.

---

# 10. Mocking Strategy

External dependencies SHALL be replaced with mocks during automated testing.

Mock implementations SHALL exist for:

- AI Providers
- MCP Servers
- Git repositories where appropriate
- File system operations where appropriate
- External APIs

Mocks SHALL produce deterministic responses.

---

# 11. Test Fixtures

Shared test fixtures SHALL be maintained within the repository.

Fixtures MAY include:

- Sample repositories
- Configuration files
- Architecture documents
- Prompt templates
- Mock provider responses
- Validation reports

Fixtures SHALL be version controlled.

---

# 12. Test Data

Test data SHALL:

- Be deterministic
- Be reusable
- Avoid production data
- Avoid sensitive information
- Be isolated from implementation code

Synthetic data SHALL be preferred over real-world datasets.

---

# 13. Test Execution

The platform SHALL support execution of individual test suites and the complete test suite.

Example commands:

```bash
pytest

pytest tests/unit

pytest tests/integration

pytest tests/e2e
```

Future CLI integration MAY provide additional convenience commands.

---

# 14. Coverage Requirements

The platform SHALL measure automated test coverage.

Coverage SHALL be reported for:

- Statements
- Branches (where supported)
- Modules
- Packages

Coverage targets SHALL be defined by the engineering governance process rather than individual implementations.

Coverage SHALL not be used as the sole measure of software quality.

---

# 15. AI Implementation Validation

Code generated through AI-assisted workflows SHALL satisfy the same testing requirements as manually written code.

AI-generated implementations SHALL:

- Pass existing tests
- Include additional tests where required
- Not reduce overall test quality
- Not bypass validation requirements

No distinction SHALL be made between AI-authored and human-authored code during validation.

---

# 16. Failure Handling

A failed automated test SHALL:

- Prevent the affected implementation from being considered complete
- Produce a readable failure report
- Identify the failing test
- Include sufficient diagnostic information for investigation

The testing framework SHALL continue executing remaining tests where practical.

---

# 17. Reporting

Test reports SHALL include:

- Execution timestamp
- Test suite executed
- Tests passed
- Tests failed
- Tests skipped
- Execution duration
- Summary status

Reports SHOULD support both human-readable and machine-readable formats.

---

# 18. Logging Requirements

The Logging Service SHALL record:

- Test execution start
- Test execution completion
- Failed test suites
- Unexpected exceptions
- Execution duration

Detailed assertion output SHALL remain within the testing framework.

---

# 19. Performance Requirements

The testing framework SHALL:

- Execute unit tests rapidly
- Support parallel execution where practical
- Avoid unnecessary external dependencies
- Support incremental execution

Performance optimization SHALL not compromise determinism.

---

# 20. CLI Integration

Future CLI commands MAY include:

```text
86vibe test

86vibe test unit

86vibe test integration

86vibe test e2e

86vibe test coverage
```

These commands SHALL provide a consistent developer experience across all repositories.

---

# 21. Governance Requirements

Testing SHALL form part of the platform Definition of Done.

Implementation SHALL NOT be considered complete unless:

- Required tests exist
- Tests pass successfully
- Existing tests continue to pass
- Documentation is updated where required

Human approval SHALL remain mandatory before architecture baselines are modified.

---

# 22. Testing Technology

The reference testing stack SHALL consist of:

| Technology | Purpose |
|------------|---------|
| `pytest` | Test framework |
| `pytest-mock` | Mocking support |
| `coverage.py` | Coverage reporting |
| `pytest-cov` | Coverage integration |

Additional tooling MAY be introduced through future ADRs.

---

# 23. Testing Lifecycle

Testing SHALL follow the lifecycle below.

```text
Implement Feature
        │
        ▼
Execute Unit Tests
        │
        ▼
Execute Integration Tests
        │
        ▼
Execute End-to-End Tests
        │
        ▼
Generate Reports
        │
        ▼
Review Results
```

Testing SHALL precede completion of implementation activities.

---

# 24. Definition of Done

The Testing Strategy SHALL be considered complete when:

- Standard test categories are implemented
- Repository structure is established
- Mocking strategy is implemented
- Test reporting is available
- Coverage reporting is supported
- CLI integration requirements are defined
- Automated tests execute successfully
- Documentation is complete
