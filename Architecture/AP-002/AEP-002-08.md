# 1. Purpose

This document defines the architecture for integrating the Model Context Protocol (MCP) into the 86-vibe platform.

The MCP Integration Layer enables AI providers to securely interact with external tools and services through a standardized protocol while maintaining the platform's architecture-first, vendor-neutral, and governance-driven principles.

The integration SHALL isolate all MCP-specific functionality behind a common platform interface so that AI providers, tools, and clients remain independently evolvable.

---

# 2. Scope

This specification defines:

- MCP architecture
- MCP server management
- MCP client architecture
- Tool discovery
- Tool invocation
- Configuration
- Security
- Error handling
- Logging
- Diagnostics
- Testing requirements

This document does not define individual MCP server implementations or third-party tool behavior.

---

# 3. Design Principles

The MCP Integration Layer SHALL adhere to the following principles.

## 3.1 Vendor Neutrality

MCP support SHALL operate independently of any AI provider.

## 3.2 Protocol Isolation

No platform component outside the MCP Integration Layer SHALL communicate directly with MCP servers.

## 3.3 Configuration Driven

Available MCP servers SHALL be determined entirely through configuration.

## 3.4 Secure by Default

Tool execution SHALL require explicit configuration and SHALL never expose secrets.

## 3.5 Extensible

New MCP servers SHALL be added without modifying existing platform components.

---

# 4. Architectural Overview

The MCP Integration Layer SHALL isolate all communication with MCP servers.

```text
                   CLI
                    │
                    ▼
            AI Service Layer
                    │
                    ▼
         MCP Integration Layer
                    │
      ┌─────────────┼─────────────┐
      │             │             │
      ▼             ▼             ▼
 Google Search   GitHub MCP   Filesystem MCP
    Server          Server         Server
```

Future servers SHALL integrate through the MCP Integration Layer.

---

# 5. Responsibilities

The MCP Integration Layer SHALL be responsible for:

- Server discovery
- Server lifecycle management
- Tool discovery
- Tool invocation
- Request validation
- Response normalization
- Connection management
- Error normalization
- Diagnostics
- Logging

It SHALL NOT:

- Execute business logic
- Perform AI prompt generation
- Manage repository workflows
- Modify architecture documents

---

# 6. MCP Client

The platform SHALL implement a single MCP Client responsible for communicating with MCP servers.

Responsibilities include:

- Starting server processes
- Establishing protocol connections
- Monitoring server health
- Sending requests
- Receiving responses
- Graceful shutdown

Platform components SHALL interact only with the MCP Client.

---

# 7. MCP Server Registry

The platform SHALL maintain a registry of configured MCP servers.

Example:

```text
MCP Registry
     │
     ├── Google Search
     ├── GitHub
     ├── Filesystem
     ├── Jira
     ├── Docker
     └── Custom Servers
```

Servers SHALL be registered during platform initialization.

---

# 8. Server Configuration

Servers SHALL be configured using project configuration.

Example:

```yaml
mcp:
  enabled: true

  servers:
    github:
      enabled: true

    filesystem:
      enabled: true

    google-search:
      enabled: false
```

The exact provider-specific configuration SHALL remain implementation specific.

---

# 9. Server Lifecycle

Each configured server SHALL follow the lifecycle below.

```text
Read Configuration
        │
        ▼
Validate Configuration
        │
        ▼
Start Server
        │
        ▼
Initialize Connection
        │
        ▼
Discover Tools
        │
        ▼
Ready
        │
        ▼
Shutdown
```

Servers that fail initialization SHALL not prevent the platform from starting unless explicitly required.

---

# 10. Tool Discovery

The MCP Client SHALL automatically discover tools exposed by connected servers.

Discovered tool metadata SHALL include:

- Tool name
- Description
- Input schema
- Output schema
- Server identifier

Tool discovery SHALL occur during server initialization.

---

# 11. Tool Invocation

All tool execution SHALL pass through the MCP Integration Layer.

Execution lifecycle:

```text
Receive Tool Request
        │
        ▼
Validate Tool
        │
        ▼
Validate Parameters
        │
        ▼
Invoke MCP Server
        │
        ▼
Receive Response
        │
        ▼
Normalize Result
        │
        ▼
Return Response
```

---

# 12. Response Normalization

Responses returned by MCP servers SHALL be normalized before exposure to platform components.

Normalized responses SHALL contain:

- Tool identifier
- Server identifier
- Execution status
- Response payload
- Execution duration
- Error information (if applicable)

Server-specific response objects SHALL NOT be exposed outside the integration layer.

---

# 13. Error Handling

MCP-specific failures SHALL be translated into platform-standard error categories.

Supported categories SHALL include:

| Error Category |
|----------------|
| Configuration Error |
| Connection Error |
| Tool Not Found |
| Validation Error |
| Timeout Error |
| Server Error |
| Internal Error |

Platform components SHALL never depend upon server-specific exception types.

---

# 14. Connection Management

The MCP Client SHALL:

- Reuse active connections where supported
- Detect unexpected disconnections
- Attempt graceful recovery where appropriate
- Cleanly terminate server processes

Connection management SHALL be transparent to calling components.

---

# 15. Security Requirements

The MCP Integration Layer SHALL:

- Execute only configured servers
- Reject unknown servers
- Validate tool parameters
- Prevent execution of unauthorized tools
- Redact sensitive values from logs
- Never expose environment secrets

The platform SHALL not automatically trust third-party MCP servers.

---

# 16. Configuration Validation

During startup and diagnostic operations, the platform SHALL validate:

- Server definitions
- Executable paths
- Required environment variables
- Configuration syntax
- Tool discovery
- Connectivity

Validation failures SHALL be reported through the CLI.

---

# 17. CLI Integration

The following commands SHALL integrate with the MCP subsystem.

```text
86vibe mcp list

86vibe mcp validate

86vibe mcp doctor
```

Future commands MAY include:

```text
86vibe mcp tools

86vibe mcp status

86vibe mcp inspect
```

---

# 18. Logging Requirements

The Logging Service SHALL record:

- Server startup
- Server shutdown
- Tool discovery
- Tool execution
- Validation failures
- Connection failures

Sensitive data SHALL always be redacted.

---

# 19. Diagnostics

The MCP subsystem SHALL support diagnostic reporting.

Diagnostic information SHALL include:

- Configured servers
- Active servers
- Failed servers
- Discovered tools
- Configuration issues
- Connectivity status

Diagnostic output SHALL not expose secrets.

---

# 20. Performance Requirements

The MCP Integration Layer SHALL:

- Minimize startup latency
- Reuse connections where practical
- Support multiple concurrent tool invocations
- Avoid unnecessary server restarts

Future connection pooling MAY be introduced without architectural changes.

---

# 21. Extensibility

The platform SHALL support future MCP servers without modification to existing platform components.

Examples include:

- Git providers
- Documentation systems
- Issue trackers
- CI/CD platforms
- Cloud providers
- Database tools
- Vector databases

Future integrations SHALL conform to this architecture.

---

# 22. Testing Requirements

Automated tests SHALL verify:

- Configuration validation
- Server registration
- Server initialization
- Tool discovery
- Tool invocation
- Response normalization
- Error normalization
- Connection failures
- Diagnostics
- Logging
- Secret redaction

Testing SHALL include mock MCP servers to avoid dependency upon external services.

---

# 23. Definition of Done

The MCP Integration Layer SHALL be considered complete when:

- All MCP communication occurs through the integration layer
- Tool discovery functions correctly
- Tool invocation is standardized
- Responses are normalized
- Errors are normalized
- Configuration validation succeeds
- Diagnostic commands operate correctly
- Automated tests pass
- Documentation is complete
