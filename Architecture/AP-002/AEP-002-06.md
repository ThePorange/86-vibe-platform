# 1. Purpose

This document defines the logging, diagnostics, telemetry, and troubleshooting architecture for the 86-vibe platform.

The objective is to provide a consistent and deterministic mechanism for recording platform activity, diagnosing failures, and supporting development without exposing sensitive information.

The logging subsystem SHALL support all platform repositories and SHALL be reusable by future components.

---

# 2. Scope

This specification defines:

- Logging architecture
- Logging levels
- Log destinations
- Log formats
- Console output
- Diagnostic commands
- Error reporting
- Exception handling
- Log rotation
- Sensitive data handling
- Future telemetry integration

This document does not define centralized telemetry or cloud-based monitoring services.

---

# 3. Design Principles

The logging subsystem SHALL adhere to the following principles.

## 3.1 Human Readable

Logs SHALL be readable using standard text editors.

## 3.2 Machine Parsable

Logs SHALL support structured formats for future automation.

## 3.3 Deterministic

Equivalent operations SHALL generate equivalent log events.

## 3.4 Secure

Sensitive information SHALL never be written to logs.

## 3.5 Minimal by Default

Normal operation SHALL produce concise output while allowing detailed diagnostics when requested.

---

# 4. Logging Architecture

All platform components SHALL use the common logging subsystem.

```text
CLI Commands
      │
      ▼
Logging Service
      │
 ┌────┴────────────┐
 │                 │
 ▼                 ▼
Console         Log File
```

Application code SHALL NOT write directly to log files.

All logging SHALL occur through the Logging Service.

---

# 5. Log Locations

Project logs SHALL be stored within the project metadata directory.

```text
.86vibe/
└── logs/
    ├── platform.log
    ├── error.log
    └── debug.log
```

The Logging Service SHALL create the directory if it does not exist.

---

# 6. Log Levels

The platform SHALL support the following logging levels.

| Level | Purpose |
|--------|---------|
| DEBUG | Detailed diagnostic information |
| INFO | Normal platform activity |
| WARNING | Recoverable issues |
| ERROR | Operation failures |
| CRITICAL | Fatal platform errors |

Logging levels SHALL follow standard Python logging semantics.

---

# 7. Default Logging Configuration

Unless explicitly overridden, the platform SHALL use:

| Setting | Default |
|----------|---------|
| Console Level | INFO |
| File Level | DEBUG |
| Error Log | Enabled |
| Debug Log | Disabled |
| Colored Console Output | Enabled |

---

# 8. Console Output

Console output SHALL prioritize developer readability.

Successful operations SHOULD display concise confirmation messages.

Example:

```text
✓ Repository validated successfully.
```

Warnings SHOULD be highlighted.

Example:

```text
⚠ Architecture document numbering is incomplete.
```

Errors SHALL clearly identify corrective action.

Example:

```text
ERROR

Missing configuration file:

.86vibe/config.yaml

Run:

86vibe init
```

---

# 9. Log File Format

Each log entry SHALL contain:

- Timestamp
- Log level
- Component
- Command
- Message

Example:

```text
2026-07-09T14:12:51Z INFO cli.init Repository initialized successfully
```

The timestamp SHALL use ISO-8601 UTC format.

---

# 10. Structured Logging

The Logging Service SHALL internally support structured log events.

Each event SHALL contain equivalent information to:

```json
{
  "timestamp": "...",
  "level": "INFO",
  "component": "cli",
  "command": "doctor",
  "message": "Environment validation complete"
}
```

The implementation MAY expose structured logging in future releases.

---

# 11. Error Logging

Unexpected failures SHALL be written to:

```text
.86vibe/logs/error.log
```

Each entry SHALL include:

- Timestamp
- Exception type
- Error message
- Stack trace (DEBUG mode only)

---

# 12. Debug Logging

Debug logging SHALL be disabled by default.

When enabled, debug logging SHALL include:

- Configuration loading
- Command execution
- Validation steps
- Provider selection
- MCP communication
- File discovery

Debug logging SHALL never expose secrets.

---

# 13. Secret Redaction

The Logging Service SHALL automatically redact:

- API keys
- Passwords
- Tokens
- Private keys
- Certificates
- Authentication headers

Example:

```text
Authorization: ********
```

No platform component SHALL bypass secret redaction.

---

# 14. Exception Handling

Unhandled exceptions SHALL be intercepted by the platform.

The Logging Service SHALL:

- Record the exception
- Display a concise user message
- Return the appropriate exit code
- Preserve detailed diagnostics in the log

Raw Python stack traces SHALL NOT be displayed unless debug mode is enabled.

---

# 15. Diagnostic Command Support

The following CLI commands SHALL integrate with the Logging Service.

```text
86vibe doctor

86vibe repo validate

86vibe config validate

86vibe mcp doctor
```

Diagnostic output SHALL summarize:

- Passed checks
- Warnings
- Failures
- Suggested corrective actions

---

# 16. Log Rotation

The platform SHALL automatically rotate log files.

Minimum requirements:

- Preserve the current log
- Archive previous logs
- Limit total retained log files
- Prevent unlimited growth

The implementation details MAY vary by operating system.

---

# 17. Performance Requirements

Logging SHALL introduce negligible overhead.

Normal command execution SHALL not perform excessive disk writes.

The Logging Service SHALL buffer writes where appropriate.

---

# 18. Component Identification

Each log entry SHALL identify the originating component.

Examples:

| Component |
|-----------|
| cli |
| config |
| repository |
| architecture |
| prompt |
| ai |
| mcp |
| logging |

Future components SHALL follow the same convention.

---

# 19. Diagnostic Report Generation

The platform SHALL support generation of a diagnostic report suitable for issue reporting.

Future command:

```bash
86vibe doctor --report
```

The report SHALL include:

- Platform version
- CLI version
- Python version
- Operating system
- Configuration validation results
- Repository validation results
- Installed providers
- Installed MCP servers

The report SHALL automatically redact sensitive information.

---

# 20. Logging Configuration

The following configuration SHALL be supported.

```yaml
logging:
  level: INFO
  console: true
  file: true
  debug: false
  rotation: true
```

Future configuration properties MAY be added without breaking compatibility.

---

# 21. Testing Requirements

Automated tests SHALL verify:

- Log creation
- Log rotation
- Console output
- Structured event generation
- Secret redaction
- Debug mode
- Error logging
- Exception interception
- Configuration loading
- Timestamp formatting

---

# 22. Definition of Done

The logging subsystem SHALL be considered complete when:

- All platform components use the Logging Service
- Console output is consistent
- Log files are created automatically
- Structured events are generated
- Secret redaction is verified
- Error handling is consistent
- Diagnostic reports are supported
- Automated tests pass
