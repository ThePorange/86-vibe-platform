# 1. Purpose

This document defines the directory structure for all Version 1.0 repositories.

The objective is to establish a consistent, scalable and maintainable layout that supports long-term development, AI-assisted implementation and clear separation of responsibilities.

All repositories SHALL follow the conventions defined within this document.

---

# 2. Repository Overview

The platform consists of three repositories.

```text
86-vibe-platform
86-vibe-cli
86-vibe-example
```

Each repository SHALL have a single primary responsibility as defined in AEP-001-08.

# 6. Directory Responsibilities

## architecture/

Contains approved Architecture Packages, ADRs and architectural diagrams.

No implementation code SHALL be stored in this directory.

---

## docs/

Contains user-facing and developer-facing documentation.

Documentation SHALL be authored in Markdown.

---

## standards/

Contains engineering standards adopted by the platform.

Examples include coding standards, documentation standards and testing standards.

---

## governance/

Contains governance documentation including licensing, contribution guidance and release policies.

---

## templates/

Contains reusable templates consumed by the platform.

Templates SHALL be version controlled.

---

## agents/

Contains AI agent definitions and associated prompt assets.

Agent implementation code SHALL NOT be stored here.

---

## schemas/

Contains JSON Schema definitions and other validation artefacts.

Schemas SHALL define the structure of configuration files, metadata and workflow definitions.

---

## src/

Contains production source code.

All executable platform logic SHALL reside within this directory.

---

## tests/

Contains automated test suites.

The directory SHALL mirror the structure of the corresponding source code where practical.

---

# 7. Naming Conventions

The following naming conventions SHALL apply.

| Item | Convention |
|------|------------|
| Directories | lowercase with hyphens where appropriate |
| Python packages | lowercase with underscores where required by Python |
| Markdown files | Pascal-Case with hyphens |
| JSON files | lowercase |
| YAML files | lowercase |
| Templates | descriptive names with extension preserved |

Repository root directories SHALL remain stable across releases.

---

# 8. Ownership

Each top-level directory SHALL have an identified owner.

| Directory | Primary Owner |
|-----------|---------------|
| architecture | Chief Systems Architect |
| governance | Product Owner |
| standards | Chief Systems Architect |
| templates | Technical Writer |
| agents | AI Orchestrator |
| src | Development Team |
| tests | QA Engineer |
| docs | Technical Writer |

Ownership defines responsibility for maintenance and review.

---

# 9. Repository Rules

The following rules SHALL apply to all repositories.

- No generated files SHALL be committed unless explicitly required.
- Secrets SHALL NOT be stored in the repository.
- Binary assets SHOULD be minimised.
- Documentation SHALL accompany new features.
- Source code SHALL remain under the `src/` directory.
- Tests SHALL be automated wherever practical.
- Repository structure SHALL remain consistent with this specification.

Changes to the directory structure SHALL require an approved Architecture Decision Record.

---

# 10. Acceptance Criteria

The repository implementation SHALL satisfy the following criteria.

- All repositories conform to the approved directory structure.
- Repository responsibilities are clearly separated.
- Documentation structure is implemented.
- Standards directory is present.
- Governance directory is present.
- Templates are organised by type.
- Source code resides under `src/`.
- Automated tests can be added without restructuring the repository.

---

# 11. Summary

The repository directory structure establishes a consistent and scalable foundation for the implementation of the 86-vibe platform.

By clearly separating architecture, governance, documentation, templates, source code and tests, the platform supports maintainability, collaboration and AI-assisted development.

All future implementation work SHALL conform to the repository structure defined within this document.
