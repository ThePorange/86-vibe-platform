## 1. Purpose

This document defines the implementation-ready specification for the 86-vibe Command Line Interface.

The CLI is the primary developer entry point for interacting with the 86-vibe platform from a local engineering environment. It SHALL provide a governed, repeatable, Git-first interface for initializing projects, validating architecture packages, managing prompts, invoking AI-assisted workflows, and supporting platform adoption.

---

## 2. Scope

This specification covers:

- CLI purpose and responsibilities
- CLI command structure
- Required command groups
- Configuration model
- Repository interaction model
- AI provider interaction model
- MCP interaction model
- Logging and error handling
- Security and governance requirements
- Implementation requirements for Cursor and Claude Code

This document does not define:

- Internal AI orchestration implementation details
- Full prompt library contents
- Jira integration behavior
- Reference application behavior
- Commercial licensing enforcement

Those items SHALL be defined in later AP-002 documents.

---

## 3. Design Principles

The CLI SHALL follow these principles:

1. **Architecture-first execution**  
   The CLI SHALL operate against approved architecture artifacts wherever possible.

2. **Human-governed workflows**  
   The CLI SHALL not silently modify baselined architecture without explicit user action.

3. **Git-first behavior**  
   All material changes SHALL be visible through Git.

4. **Vendor-neutral AI execution**  
   The CLI SHALL not be tightly coupled to a single AI model provider.

5. **Local-first developer experience**  
   The CLI SHALL work from a local checkout without requiring hosted platform services.

6. **Safe defaults**  
   Destructive or irreversible actions SHALL require confirmation.

---

## 4. CLI Package

The CLI SHALL be implemented in the repository:

```text
86-vibe-cli
```

The Python package name SHALL be:

```text
vibe_cli
```

The installed command SHALL be:

```text
86vibe
```

Example:

```bash
86vibe --help
```

---

## 5. CLI Technology Requirements

The CLI SHALL be implemented using Python.

The preferred CLI framework SHALL be:

```text
Typer
```

The CLI MAY use Click indirectly through Typer.

The CLI SHALL use:

- `pyproject.toml` for packaging
- `ruff` for linting
- `pytest` for tests
- `pydantic` for configuration validation where required
- `rich` for formatted terminal output where useful

---

## 6. Top-Level Command Structure

The CLI SHALL expose the following top-level command groups:

```text
86vibe init
86vibe doctor
86vibe config
86vibe arch
86vibe prompt
86vibe ai
86vibe mcp
86vibe repo
86vibe version
```

Each command SHALL support:

```bash
--help
```

The CLI SHALL return non-zero exit codes on failure.

---

## 7. Command Specifications

### 7.1 `86vibe init`

Initializes 86-vibe metadata in a repository.

Example:

```bash
86vibe init
```

Responsibilities:

- Detect current Git repository
- Create `.86vibe/` if missing
- Create default CLI configuration
- Validate expected documentation folders
- Report initialization status

Required output:

```text
86-vibe initialized successfully
```

The command SHALL NOT overwrite existing configuration without confirmation.

---

### 7.2 `86vibe doctor`

Validates local environment readiness.

Example:

```bash
86vibe doctor
```

Checks SHALL include:

- Python version
- Git availability
- Current Git repository status
- Required directory structure
- 86-vibe configuration validity
- AI provider configuration presence
- MCP configuration presence, where applicable

The command SHALL produce a pass/warn/fail summary.

---

### 7.3 `86vibe config`

Manages CLI configuration.

Required subcommands:

```text
86vibe config show
86vibe config validate
86vibe config set
86vibe config unset
```

Examples:

```bash
86vibe config show
86vibe config validate
86vibe config set ai.provider openrouter
86vibe config unset ai.default_model
```

Configuration values SHALL be stored in local project configuration unless explicitly defined as user-level settings in a future ADR.

---

### 7.4 `86vibe arch`

Manages architecture documentation workflows.

Required subcommands:

```text
86vibe arch list
86vibe arch validate
86vibe arch status
86vibe arch baseline
```

Responsibilities:

- List architecture documents
- Validate required metadata
- Validate document numbering conventions
- Identify draft, approved, and missing documents
- Support controlled baselining of approved documents

The CLI SHALL NOT approve architecture on behalf of a human.

---

### 7.5 `86vibe prompt`

Manages prompt templates and implementation prompts.

Required subcommands:

```text
86vibe prompt list
86vibe prompt show
86vibe prompt validate
86vibe prompt render
```

Example:

```bash
86vibe prompt render implement-feature --var feature_id=CLI-001
```

Responsibilities:

- Load prompt templates
- Validate required variables
- Render prompts for AI tools
- Support reproducible AI implementation workflows

Prompt rendering SHALL be deterministic for the same inputs.

---

### 7.6 `86vibe ai`

Provides AI workflow integration.

Required subcommands:

```text
86vibe ai providers
86vibe ai models
86vibe ai plan
86vibe ai implement
```

Responsibilities:

- List configured AI providers
- List available or configured models
- Generate implementation plans from approved architecture
- Invoke AI-assisted implementation workflows

The CLI SHALL clearly distinguish between:

- Planning output
- Suggested implementation
- Actual file modifications

The CLI SHALL NOT make file changes through AI execution unless the user explicitly invokes an implementation command.

---

### 7.7 `86vibe mcp`

Manages MCP-related configuration and diagnostics.

Required subcommands:

```text
86vibe mcp list
86vibe mcp validate
86vibe mcp doctor
```

Responsibilities:

- List configured MCP servers
- Validate MCP configuration structure
- Detect missing command paths or environment variables
- Report MCP readiness

The CLI SHALL NOT expose secrets in terminal output.

---

### 7.8 `86vibe repo`

Provides repository utilities.

Required subcommands:

```text
86vibe repo status
86vibe repo validate
86vibe repo docs
```

Responsibilities:

- Validate repository structure
- Confirm required documentation folders
- Confirm Git working tree state
- Report documentation inventory

The CLI SHALL NOT perform Git commits unless a future approved document explicitly defines that behavior.

---

### 7.9 `86vibe version`

Displays CLI version information.

Example:

```bash
86vibe version
```

Required output:

- CLI version
- Python version
- Platform name
- Optional Git commit hash, if available

## 8. Configuration Model

The CLI SHALL support a project-local configuration file:

```text
.86vibe/config.yaml
```

Minimum configuration schema:

```yaml
platform:
  name: 86-vibe

repository:
  type: platform
  authoritative: true

ai:
  provider: openrouter
  default_model: null

mcp:
  enabled: true
  config_path: null

governance:
  require_human_approval: true
  allow_architecture_overwrite: false
```

Configuration validation SHALL fail if required keys are missing.

Secrets SHALL NOT be stored in `.86vibe/config.yaml`.

---

## 9. Directory Requirements

The CLI SHALL expect the following project metadata directory:

```text
.86vibe/
  config.yaml
  cache/
  logs/
```

The CLI MAY create missing non-source metadata folders during initialization.

The CLI SHALL NOT create or modify architecture documents unless explicitly invoked by the user.

---

## 10. Logging Requirements

The CLI SHALL write logs to:

```text
.86vibe/logs/
```

Logging SHALL include:

- command name
- timestamp
- result status
- error details, where applicable

Logs SHALL NOT include:

- API keys
- access tokens
- secrets
- full prompt contents containing sensitive values

---

## 11. Error Handling

The CLI SHALL use consistent error categories:

```text
CONFIG_ERROR
REPO_ERROR
VALIDATION_ERROR
AI_PROVIDER_ERROR
MCP_ERROR
USER_CANCELLED
INTERNAL_ERROR
```

User-facing errors SHALL be readable and actionable.

Example:

```text
VALIDATION_ERROR: Missing required file .86vibe/config.yaml.
Run: 86vibe init
```

---

## 12. Exit Codes

The CLI SHALL use the following exit codes:

```text
0   success
1   general failure
2   validation failure
3   configuration failure
4   repository failure
5   AI provider failure
6   MCP failure
130 user cancelled
```

---

## 13. Security Requirements

The CLI SHALL:

- Never print secrets by default
- Redact environment variables that appear to contain credentials
- Require confirmation before destructive actions
- Avoid sending repository contents to AI providers unless explicitly requested
- Prefer scoped file inputs for AI workflows
- Record enough metadata for audit without storing sensitive payloads

---

## 14. Governance Requirements

The CLI SHALL enforce the following governance rules:

1. Approved architecture documents are treated as baselined.
2. Baselined documents SHALL NOT be modified by default.
3. Architecture validation SHALL identify missing or malformed documents.
4. Human approval SHALL remain outside automated CLI decision-making.
5. Any future automated approval workflow SHALL require a separate ADR.

---

## 15. Implementation Requirements

Cursor and Claude Code SHALL implement the CLI in small, testable increments.

Minimum implementation sequence:

1. Create Python package skeleton
2. Implement top-level Typer application
3. Implement `version`
4. Implement `init`
5. Implement configuration loading and validation
6. Implement `doctor`
7. Implement `repo status`
8. Implement `arch list`
9. Implement tests for each command
10. Add README usage examples

---

## 16. Testing Requirements

The CLI SHALL include automated tests for:

- command availability
- help output
- config creation
- config validation
- missing Git repository handling
- missing configuration handling
- invalid configuration handling
- exit codes
- secret redaction behavior

Tests SHALL be runnable with:

```bash
pytest
```

---

## 17. Definition of Done

A CLI implementation increment SHALL be considered complete when:

- Commands are implemented according to this specification
- Tests pass locally
- Help output is available for all commands
- Configuration validation is implemented
- Errors are readable and actionable
- No secrets are printed in test output
- Documentation examples are updated
- Changes are committed through normal Git workflow
