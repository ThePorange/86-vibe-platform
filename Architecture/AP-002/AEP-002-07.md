# 1. Purpose

This document defines the architecture for the AI Provider Abstraction Layer (APAL), which provides a vendor-neutral interface between the 86-vibe platform and external AI providers.

The abstraction layer ensures that all platform components communicate with AI providers through a single, governed interface, allowing providers and models to be changed without requiring changes to the remainder of the platform.

This architecture is fundamental to achieving the vendor-neutral design principle established in AP-001.

---

# 2. Scope

This specification defines:

- AI Provider architecture
- Provider abstraction model
- Provider lifecycle
- Model selection
- Request lifecycle
- Response normalization
- Provider registration
- Provider capabilities
- Error handling
- Retry behavior
- Streaming support
- Testing requirements

This document does not define provider-specific implementations or prompt engineering techniques.

---

# 3. Design Principles

The AI Provider Abstraction Layer SHALL adhere to the following principles.

## 3.1 Vendor Neutrality

No component outside the abstraction layer SHALL depend upon a specific AI provider SDK or API.

## 3.2 Single Interface

All AI interactions SHALL occur through a common interface.

## 3.3 Extensibility

New providers SHALL be added without modifying existing platform components.

## 3.4 Deterministic Configuration

Provider selection SHALL be driven entirely through configuration.

## 3.5 Testability

The abstraction layer SHALL support mock providers for automated testing.

---

# 4. Architectural Overview

The AI Provider Abstraction Layer SHALL isolate all provider-specific logic.

```text
                CLI
                 │
                 ▼
        AI Service Layer
                 │
                 ▼
     AI Provider Abstraction Layer
                 │
     ┌───────────┼────────────┐
     │           │            │
     ▼           ▼            ▼
 OpenRouter   Anthropic    OpenAI
 Provider     Provider     Provider
```

Future providers SHALL connect only through the abstraction layer.

---

# 5. Responsibilities

The AI Provider Abstraction Layer SHALL be responsible for:

- Provider selection
- Model selection
- Request validation
- Request execution
- Response normalization
- Error normalization
- Retry handling
- Timeout handling
- Streaming support
- Capability discovery

It SHALL NOT perform:

- Prompt generation
- Business logic
- Architecture validation
- Repository operations

---

# 6. Provider Interface

Every provider implementation SHALL expose equivalent capabilities.

Required operations include:

- Initialize provider
- Validate configuration
- List available models
- Execute request
- Execute streaming request
- Report capabilities
- Shutdown provider

The exact programming interface SHALL be determined during implementation.

---

# 7. Provider Registry

The platform SHALL maintain a Provider Registry responsible for discovering available providers.

Example architecture:

```text
Provider Registry
        │
        ├── OpenRouter
        ├── Anthropic
        ├── OpenAI
        ├── Local LLM
        └── Mock Provider
```

The registry SHALL expose only providers that successfully initialize.

---

# 8. Supported Providers

The initial platform SHALL support:

| Provider | Required |
|-----------|----------|
| OpenRouter | Yes |
| Anthropic | Yes |
| OpenAI | Yes |
| Mock Provider | Yes |

Additional providers MAY be added in future releases.

---

# 9. Provider Selection

Provider selection SHALL be determined using configuration.

Example:

```yaml
ai:
  provider: openrouter
```

CLI overrides SHALL take precedence.

Example:

```bash
86vibe ai plan --provider anthropic
```

---

# 10. Model Selection

Models SHALL be selected independently of providers.

Example:

```yaml
ai:
  provider: openrouter
  default_model: anthropic/claude-opus-4
```

The abstraction layer SHALL validate that the selected model is supported by the configured provider.

---

# 11. Request Lifecycle

Every AI request SHALL follow the same lifecycle.

```text
Receive Request
        │
        ▼
Validate Configuration
        │
        ▼
Select Provider
        │
        ▼
Select Model
        │
        ▼
Validate Request
        │
        ▼
Execute Provider Call
        │
        ▼
Normalize Response
        │
        ▼
Return Result
```

---

# 12. Request Structure

All provider requests SHALL be internally normalized.

Minimum request attributes SHALL include:

- Provider
- Model
- Prompt
- Temperature
- Maximum tokens
- Streaming flag
- Metadata

Future attributes MAY be added without breaking compatibility.

---

# 13. Response Normalization

Responses returned by providers SHALL be normalized before being exposed to platform components.

Normalized responses SHALL include:

- Generated content
- Model identifier
- Provider identifier
- Token usage (if available)
- Finish reason
- Execution duration

Provider-specific response objects SHALL NOT be exposed outside the abstraction layer.

---

# 14. Provider Capabilities

Each provider SHALL report its supported capabilities.

Examples include:

- Chat completion
- Streaming
- Tool calling
- Structured output
- Image generation
- Function calling

Capabilities SHALL be discoverable at runtime.

---

# 15. Error Normalization

Provider-specific exceptions SHALL be translated into platform-standard errors.

Supported error categories SHALL include:

| Error Category |
|----------------|
| Authentication Error |
| Authorization Error |
| Configuration Error |
| Rate Limit Error |
| Timeout Error |
| Network Error |
| Provider Error |
| Validation Error |
| Internal Error |

Platform components SHALL never receive provider-specific exception types.

---

# 16. Retry Strategy

Retry behavior SHALL be configurable.

Retry SHALL be considered only for transient failures.

Examples include:

- Network interruption
- Temporary provider outage
- Rate limiting
- Service unavailable

Authentication failures SHALL NOT be retried.

---

# 17. Timeout Handling

Provider requests SHALL support configurable timeouts.

Example configuration:

```yaml
ai:
  timeout: 60
```

Expired requests SHALL return a standardized timeout error.

---

# 18. Streaming Support

The abstraction layer SHALL support streaming responses where supported by the provider.

Streaming SHALL remain optional.

Consumers SHALL not be required to support streaming.

---

# 19. Mock Provider

The platform SHALL include a Mock Provider.

The Mock Provider SHALL:

- Require no network access
- Produce deterministic responses
- Support automated testing
- Simulate provider failures
- Simulate latency

The Mock Provider SHALL be the preferred provider for unit testing.

---

# 20. Security Requirements

The abstraction layer SHALL:

- Never expose API keys
- Never log prompts containing secrets
- Validate provider credentials before execution
- Reject unsupported providers
- Reject unsupported models

Provider credentials SHALL remain external to project configuration.

---

# 21. Performance Requirements

The abstraction layer SHALL:

- Minimize startup overhead
- Reuse provider connections where practical
- Avoid unnecessary provider initialization
- Support future request caching

Caching behavior SHALL be defined in a future architecture package.

---

# 22. Testing Requirements

Automated tests SHALL verify:

- Provider discovery
- Provider initialization
- Configuration validation
- Model validation
- Request execution
- Response normalization
- Error normalization
- Retry behavior
- Timeout handling
- Mock Provider functionality
- Streaming support

Network-independent tests SHALL use the Mock Provider.

---

# 23. Definition of Done

The AI Provider Abstraction Layer SHALL be considered complete when:

- All provider interactions occur through the abstraction layer
- Provider selection is configuration-driven
- Responses are normalized
- Errors are normalized
- Mock Provider is implemented
- Streaming is supported
- Automated tests pass
- Documentation is complete
