# 1. Purpose

This document defines the Repository Services Layer (RSL) for the 86-vibe platform.

The Repository Services Layer provides a unified, platform-independent interface for interacting with Git repositories and repository contents. It encapsulates repository discovery, validation, file operations, metadata management, and repository state inspection behind a governed service layer.

This architecture ensures that all platform components interact with repositories consistently while maintaining the Git-first engineering principles established in AP-001.

---

# 2. Scope

This specification defines:

- Repository Services architecture
- Repository discovery
- Repository validation
- Repository metadata
- File system abstraction
- Document discovery
- Repository state inspection
- Git integration
- Repository diagnostics
- Testing requirements

This document does not define Git workflow policies or CI/CD pipelines.

---

# 3. Design Principles

The Repository Services Layer SHALL adhere to the following principles.

## 3.1 Git First

All repository operations SHALL assume Git as the authoritative source of project history.

## 3.2 Repository Agnostic

The Repository Services Layer SHALL operate independently of repository hosting providers.

## 3.3 Read-First Architecture

Repository inspection SHALL be read-only by default.

## 3.4 Consistent Access

All platform components SHALL access repository information through a common service layer.

## 3.5 Deterministic

Repository discovery and metadata generation SHALL produce consistent results for identical repository states.

---

# 4. Architectural Overview

The Repository Services Layer SHALL isolate repository operations from platform business logic.

```text
                 CLI
                  │
                  ▼
       Repository Services Layer
                  │
      ┌───────────┼────────────┐
      │           │            │
      ▼           ▼            ▼
 Repository   File System     Git
 Discovery     Services     Operations
```

No platform component SHALL communicate directly with Git or the file system outside the Repository Services Layer unless explicitly approved through an ADR.

---

# 5. Responsibilities

The Repository Services Layer SHALL be responsible for:

- Repository discovery
- Repository validation
- Repository metadata
- Git state inspection
- File discovery
- Document discovery
- Directory validation
- Repository diagnostics
- Read-only file access

The Repository Services Layer SHALL NOT:

- Execute AI workflows
- Render prompts
- Modify architecture documents
- Approve repository changes
- Execute Git commits

---

# 6. Repository Discovery

The Repository Services Layer SHALL automatically identify the active repository.

Repository discovery SHALL verify:

- Presence of a Git repository
- Repository root
- Platform metadata
- Required directory structure

Discovery SHALL fail if no valid repository is detected.

---

# 7. Repository Metadata

The Repository Services Layer SHALL expose normalized repository metadata.

Minimum metadata SHALL include:

| Property | Description |
|----------|-------------|
| Repository Name | Repository identifier |
| Repository Root | Absolute path |
| Repository Type | Platform, CLI, Example, or Future |
| Current Branch | Active Git branch |
| Default Branch | Repository default branch |
| Git Status | Clean or Modified |
| Platform Version | Current platform version |

Additional metadata MAY be introduced in future releases.

---

# 8. Repository Types

The platform SHALL recognize the following repository classifications.

| Repository Type | Purpose |
|----------------|---------|
| Platform | Primary architecture and implementation repository |
| CLI | Command-line interface implementation |
| Example | Reference implementation |
| Extension | Future platform extensions |
| Plugin | Future third-party plugins |

Unknown repository types SHALL generate warnings but SHALL NOT prevent operation unless required by the current command.

---

# 9. Directory Validation

The Repository Services Layer SHALL validate required repository directories.

Minimum platform repository structure:

```text
docs/
architecture/
prompts/
src/
tests/
```

Repository-specific validation rules MAY be defined by future architecture documents.

---

# 10. File Discovery

The Repository Services Layer SHALL provide deterministic file discovery.

Supported capabilities SHALL include:

- Recursive file discovery
- File filtering
- Extension filtering
- Directory exclusion
- Hidden file handling

The ordering of discovered files SHALL be deterministic.

---

# 11. Architecture Document Discovery

The Repository Services Layer SHALL discover architecture documents automatically.

Discovered metadata SHALL include:

- Document identifier
- Document title
- Package
- Status
- File path

Document validation SHALL be performed by the Document Validation Framework defined in a future architecture document.

---

# 12. Git Integration

The Repository Services Layer SHALL expose read-only Git functionality.

Supported operations SHALL include:

- Current branch
- Repository status
- Latest commit
- Repository root
- Current tag
- Repository cleanliness

Write operations SHALL require future architectural approval.

---

# 13. Repository Validation

Repository validation SHALL include:

- Git repository presence
- Required directories
- Required files
- Platform configuration
- Architecture directory
- Prompt directory
- Python project configuration

Validation SHALL be available through CLI commands.

---

# 14. Repository Status Model

The Repository Services Layer SHALL classify repositories using the following status model.

| Status | Meaning |
|---------|---------|
| Valid | Repository passes validation |
| Warning | Repository usable with minor issues |
| Invalid | Repository cannot be used |
| Unknown | Validation incomplete |

Commands MAY determine whether warnings constitute failures.

---

# 15. CLI Integration

The following commands SHALL integrate with the Repository Services Layer.

```text
86vibe repo status

86vibe repo validate

86vibe repo docs

86vibe doctor
```

Future commands MAY include:

```text
86vibe repo inspect

86vibe repo inventory

86vibe repo summary
```

---

# 16. Logging Requirements

The Logging Service SHALL record:

- Repository discovery
- Validation results
- Directory validation
- Git inspection
- Repository classification
- Validation failures

Repository contents SHALL NOT be logged unless explicitly enabled for debugging.

---

# 17. Security Requirements

The Repository Services Layer SHALL:

- Operate within the active repository boundary
- Prevent unintended access outside the repository root
- Normalize file paths
- Prevent directory traversal
- Avoid exposing secrets discovered within repository files

The Repository Services Layer SHALL not inspect ignored files unless explicitly requested.

---

# 18. Performance Requirements

Repository discovery SHALL minimize unnecessary filesystem traversal.

The Repository Services Layer SHALL:

- Cache repository metadata
- Cache directory validation results
- Reuse Git metadata where practical

Caches SHALL be invalidated whenever repository state changes.

---

# 19. Extensibility

Future repository capabilities MAY include:

- Multi-repository workspaces
- Repository dependency graphs
- Cross-repository validation
- Workspace inventory
- Monorepo support
- Remote repository inspection

These capabilities SHALL be introduced without breaking existing interfaces.

---

# 20. Testing Requirements

Automated tests SHALL verify:

- Repository discovery
- Repository validation
- Repository classification
- Directory validation
- File discovery
- Git inspection
- Metadata generation
- Invalid repository handling
- Repository caching
- Path normalization

Testing SHALL include representative Platform, CLI, and Example repositories.

---

# 21. Definition of Done

The Repository Services Layer SHALL be considered complete when:

- Repository discovery functions correctly
- Validation succeeds
- Metadata is normalized
- Git inspection operates correctly
- CLI integration is complete
- Repository boundaries are enforced
- Automated tests pass
- Documentation is complete
