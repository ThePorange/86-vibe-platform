# 1. Purpose

This document defines the Docker Development Environment architecture for the 86-vibe platform.

The Docker Development Environment provides a consistent, reproducible, isolated development runtime that allows contributors, AI engineering agents, and future Continuous Integration systems to execute the platform in an identical environment regardless of the underlying operating system.

The objective is to eliminate environment drift while maintaining a lightweight local development experience.

---

# 2. Scope

This specification defines:

- Docker architecture
- Development container strategy
- Container responsibilities
- Image management
- Volume management
- Networking
- Environment configuration
- CLI integration
- Security
- Testing requirements

This document does not define production deployment containers or Kubernetes deployment architecture.

---

# 3. Design Principles

The Docker Development Environment SHALL adhere to the following principles.

## 3.1 Reproducible

Every developer SHALL be able to create an identical environment using the same source repository.

## 3.2 Lightweight

Containers SHALL contain only the software required for development.

## 3.3 Repository Independent

Each repository SHALL be executable within the same development environment architecture.

## 3.4 Disposable

Development containers SHALL be treated as ephemeral resources.

## 3.5 Vendor Neutral

The architecture SHALL not depend upon Docker Desktop-specific functionality.

---

# 4. Architectural Overview

The Docker Development Environment SHALL isolate the host operating system from platform execution.

```text
               Developer
                    │
                    ▼
         Docker Development Environment
                    │
         ┌──────────┼──────────┐
         │          │          │
         ▼          ▼          ▼
     Platform     CLI      Example App
                    │
                    ▼
          Shared Development Services
```

Each repository SHALL execute independently while sharing common development services where appropriate.

---

# 5. Responsibilities

The Docker Development Environment SHALL be responsible for:

- Development runtime
- Dependency isolation
- Python execution
- CLI execution
- Development tooling
- Volume mounting
- Environment configuration

The Docker Development Environment SHALL NOT:

- Replace local Git repositories
- Deploy production workloads
- Perform release publishing
- Replace repository version control

---

# 6. Container Architecture

The development environment SHALL consist of a primary development container.

Future supporting containers MAY include:

| Container | Purpose |
|-----------|---------|
| Development | Primary development environment |
| Documentation | Documentation generation |
| Testing | Isolated test execution |
| Services | Supporting development services |

Additional containers MAY be introduced through future architecture packages.

---

# 7. Base Image

The development image SHALL:

- Use an official Python base image
- Support the platform's minimum Python version
- Be actively maintained
- Minimize unnecessary packages

The exact image tag SHALL be defined during implementation.

---

# 8. Development Workspace

The repository SHALL be mounted into the container as the working directory.

Example:

```text
/workspace/
```

The container SHALL execute from the repository root.

---

# 9. Volume Management

Persistent volumes SHALL be limited to development artifacts.

Typical mounted locations include:

```text
workspace/
.86vibe/
```

Temporary build artifacts SHALL remain inside the container unless explicitly exported.

---

# 10. Environment Variables

Environment variables SHALL provide runtime configuration.

Examples include:

```text
OPENROUTER_API_KEY

ANTHROPIC_API_KEY

OPENAI_API_KEY

VIBE_LOG_LEVEL
```

Secrets SHALL remain external to container images.

---

# 11. Development Tools

The development image SHALL include the standard engineering toolchain.

Minimum tools include:

| Tool | Purpose |
|------|---------|
| Python | Platform runtime |
| Git | Version control |
| pip | Package management |
| pytest | Testing |
| Ruff | Linting |
| Typer | CLI framework support |

Additional tooling MAY be added through future ADRs.

---

# 12. Networking

The development environment SHALL:

- Support outbound internet access
- Support communication with configured MCP servers
- Support AI provider connectivity
- Avoid exposing unnecessary ports

Network configuration SHALL remain minimal by default.

---

# 13. Repository Integration

Repositories SHALL execute directly from mounted source code.

The Docker environment SHALL NOT require copying source files into container images during normal development.

Source code changes SHALL be immediately visible inside the running container.

---

# 14. Build Integration

The Docker Development Environment SHALL support execution of:

- Unit tests
- Integration tests
- CLI commands
- Build processes
- Documentation validation

The container SHALL behave consistently with local execution.

---

# 15. CLI Integration

Future CLI commands MAY include:

```text
86vibe docker build

86vibe docker shell

86vibe docker start

86vibe docker stop

86vibe docker clean
```

These commands SHALL simplify developer interaction with the Docker environment.

---

# 16. Logging Requirements

The Logging Service SHALL record:

- Container initialization
- Environment validation
- CLI execution
- Container shutdown
- Startup failures

Container runtime logs SHALL remain separate from platform logs.

---

# 17. Security Requirements

The Docker Development Environment SHALL:

- Execute as a non-root user where practical
- Avoid embedding secrets
- Minimize container privileges
- Restrict unnecessary capabilities
- Mount only required directories

Container images SHALL not contain project credentials.

---

# 18. Performance Requirements

The Docker Development Environment SHALL:

- Start quickly
- Reuse cached dependencies
- Minimize image size
- Avoid unnecessary rebuilds

Performance optimizations SHALL not compromise reproducibility.

---

# 19. Configuration

Container configuration SHALL remain external to the image.

Configuration SHALL include:

- Environment variables
- Mounted volumes
- Working directory
- Optional development settings

Repository-specific configuration SHALL remain within the repository.

---

# 20. Testing Requirements

Automated tests SHALL verify:

- Container startup
- Python runtime
- Repository mounting
- Environment variable loading
- CLI execution
- Test execution
- Build execution
- Volume mounting
- Non-root execution

Testing SHALL verify reproducibility across supported operating systems.

---

# 21. Failure Handling

Container startup failures SHALL provide:

- Failure summary
- Validation results
- Missing dependency information
- Suggested corrective action

The platform SHALL fail gracefully when the Docker runtime is unavailable.

---

# 22. Definition of Done

The Docker Development Environment SHALL be considered complete when:

- Containers start successfully
- Repository mounting functions correctly
- Development tools are available
- CLI commands execute successfully
- Tests execute successfully
- Build processes execute successfully
- Security requirements are satisfied
- Documentation is complete
