# 1. Purpose

This document defines the Service Lifecycle & Dependency Management architecture for the 86-vibe platform.

The platform is composed of a collection of loosely coupled services that cooperate to provide a unified engineering experience. This specification defines the lifecycle, interaction model, dependency management, and operational state of every platform service.

The objective is to ensure that platform services remain modular, deterministic, independently testable, and extensible while avoiding tight coupling between components.

---

# 2. Scope

This specification defines:

- Service architecture
- Service lifecycle
- Service states
- Dependency declaration
- Dependency resolution
- Service registration
- Service discovery
- Service health
- Optional services
- Service shutdown
- Testing requirements

This document does not define plugin architecture or runtime extension loading.

---

# 3. Design Principles

The Service Lifecycle architecture SHALL adhere to the following principles.

## 3.1 Single Responsibility

Each platform service SHALL provide a clearly defined responsibility.

## 3.2 Loose Coupling

Services SHALL communicate through published interfaces rather than direct implementation dependencies.

## 3.3 Explicit Dependencies

Every service SHALL explicitly declare its dependencies.

## 3.4 Lifecycle Managed

Every service SHALL follow the same lifecycle model.

## 3.5 Independently Testable

Services SHALL be testable in isolation through mocks or test implementations.

---

# 4. Architectural Overview

Platform services SHALL be coordinated through the Bootstrap Manager.

```text
                Bootstrap Manager
                        │
        ┌───────────────┼────────────────┐
        │               │                │
        ▼               ▼                ▼
 Configuration      Logging        Repository
        │               │                │
        └──────┬────────┴────────┬───────┘
               ▼                 ▼
     Document Validation    Prompt Framework
               │                 │
               └────────┬────────┘
                        ▼
                AI Provider Layer
                        │
                        ▼
                 MCP Integration
```

No service SHALL bypass the Bootstrap Manager during initialization.

---

# 5. Service Responsibilities

Every service SHALL:

- Expose a clearly defined interface
- Declare required dependencies
- Support initialization
- Support shutdown
- Report health
- Report readiness

Services SHALL avoid direct knowledge of unrelated platform components.

---

# 6. Standard Service Lifecycle

Every service SHALL implement the following lifecycle.

```text
Constructed
      │
      ▼
Configured
      │
      ▼
Initialized
      │
      ▼
Ready
      │
      ▼
Running
      │
      ▼
Stopping
      │
      ▼
Stopped
```

State transitions SHALL be deterministic.

---

# 7. Service States

The platform SHALL recognize the following lifecycle states.

| State | Description |
|--------|-------------|
| Constructed | Service instance created |
| Configured | Configuration applied |
| Initialized | Internal initialization complete |
| Ready | Ready for use |
| Running | Active execution |
| Stopping | Shutdown in progress |
| Stopped | Shutdown complete |
| Failed | Initialization or execution failed |

Every service SHALL expose its current lifecycle state.

---

# 8. Dependency Declaration

Every service SHALL explicitly declare:

- Required dependencies
- Optional dependencies

Example:

```text
Prompt Framework

Required:
- Configuration Service
- Logging Service

Optional:
- Repository Services
```

Hidden runtime dependencies SHALL not be permitted.

---

# 9. Dependency Resolution

Dependencies SHALL be resolved before service initialization.

The Bootstrap Manager SHALL verify:

- Required services exist
- Required services are initialized
- Required services are ready

Initialization SHALL fail if mandatory dependencies are unavailable.

---

# 10. Service Registration

Services SHALL register themselves during initialization.

Registered metadata SHALL include:

- Service identifier
- Service version
- Lifecycle state
- Supported capabilities
- Dependency information

Registration SHALL occur only after successful initialization.

---

# 11. Service Discovery

Platform components SHALL discover services through the Bootstrap Manager.

Direct construction of shared platform services SHALL NOT be permitted.

Service discovery SHALL provide:

- Service availability
- Lifecycle state
- Capability information

---

# 12. Service Communication

Platform services SHALL communicate through published interfaces.

Communication SHALL avoid:

- Circular dependencies
- Global mutable state
- Direct implementation coupling

Future event-driven communication MAY be introduced through an ADR.

---

# 13. Optional Services

Some platform services MAY be optional.

Examples include:

- MCP Integration
- Future Plugin Manager
- Future Telemetry Service

Optional service failures SHALL not prevent platform startup unless explicitly configured as required.

---

# 14. Health Monitoring

Every service SHALL expose health information.

Minimum health states:

| State | Meaning |
|--------|---------|
| Healthy | Operating normally |
| Degraded | Operating with reduced capability |
| Failed | Unable to perform required responsibilities |

Health SHALL be evaluated independently of lifecycle state.

---

# 15. Readiness

Readiness SHALL indicate whether a service can safely accept requests.

A service MAY be:

- Initialized but not Ready
- Ready but not Running
- Running and Healthy

Readiness SHALL be evaluated before command execution.

---

# 16. Shutdown Sequence

Services SHALL terminate in reverse dependency order.

Example:

```text
MCP Integration
       │
       ▼
AI Provider Layer
       │
       ▼
Prompt Framework
       │
       ▼
Repository Services
       │
       ▼
Logging
       │
       ▼
Configuration
```

Shutdown SHALL release resources cleanly.

---

# 17. Failure Handling

If a service enters the Failed state:

- The failure SHALL be logged.
- Dependent services SHALL not initialize.
- The Bootstrap Manager SHALL determine whether startup may continue.

Failure handling SHALL remain deterministic.

---

# 18. Logging Requirements

The Logging Service SHALL record:

- Service construction
- Initialization
- Registration
- State transitions
- Health changes
- Shutdown
- Failures

Lifecycle events SHALL support future diagnostic tooling.

---

# 19. Performance Requirements

The service architecture SHALL:

- Minimize initialization overhead
- Avoid redundant service creation
- Reuse shared platform services
- Support future lazy initialization where appropriate

Performance optimization SHALL not compromise lifecycle correctness.

---

# 20. Testing Requirements

Automated tests SHALL verify:

- Lifecycle transitions
- Dependency declaration
- Dependency resolution
- Registration
- Discovery
- Health reporting
- Readiness reporting
- Failure handling
- Shutdown ordering

Testing SHALL include simulated dependency failures.

---

# 21. CLI Integration

The Bootstrap Manager SHALL provide service lifecycle information to future CLI commands.

Potential commands include:

```text
86vibe services list

86vibe services status

86vibe services health

86vibe services inspect
```

These commands are reserved for future implementation.

---

# 22. Definition of Done

The Service Lifecycle architecture SHALL be considered complete when:

- Service lifecycle is standardized
- Dependencies are explicit
- Registration functions correctly
- Discovery functions correctly
- Health reporting is available
- Shutdown sequencing is implemented
- Automated tests pass
- Documentation is complete
