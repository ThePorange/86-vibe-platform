# 1. Purpose

This document defines the Python package architecture for the 86-vibe platform.

The objective is to establish a modular, maintainable and extensible package structure that supports long-term evolution while remaining straightforward for both human developers and AI agents to understand.

Every package SHALL have a clearly defined responsibility.

---

# 2. Architectural Principles

The package architecture SHALL adhere to the following principles.

- Single Responsibility
- High Cohesion
- Loose Coupling
- Explicit Interfaces
- Dependency Inversion
- Testability
- Replaceability

Packages SHALL communicate through published interfaces rather than direct implementation dependencies.

---

# 3. Root Package

The root package SHALL be:

```text
vibe
```

All production code SHALL reside beneath this package.

No executable business logic SHALL exist outside the root package.

# 6. Dependency Rules

The following dependency rules SHALL apply.

```text
CLI

↓

Workflow

↓

Services

↓

Packages

↓

Models
```

Packages SHALL NOT depend upon the CLI.

Models SHALL remain independent.

Utilities SHALL NOT depend upon higher-level packages.

Circular dependencies SHALL NOT be introduced.

---

# 7. Public Interfaces

Every package SHALL expose a clearly defined public interface.

Internal implementation modules SHOULD remain private.

Consumers SHALL interact with packages through exported interfaces rather than internal modules.

---

# 8. Package Initialisation

Each package SHALL contain:

```
__init__.py
```

Package metadata SHOULD be exported through the package root.

Package initialisation SHALL remain lightweight.

---

# 9. Testing Structure

Each package SHALL have a corresponding test package.

Example:

```text
src/

    vibe/

        workflow/

tests/

    workflow/
```

Tests SHALL mirror the production package hierarchy.

---

# 10. Acceptance Criteria

The package architecture SHALL satisfy the following.

- Every package has one responsibility.
- No circular dependencies exist.
- Business logic resides within services and domain packages.
- CLI contains no business logic.
- Domain models remain reusable.
- Every package is independently testable.
- Public interfaces are documented.

---

# 11. Summary

The Python Package Architecture establishes the modular implementation structure for the 86-vibe platform.

By organising the platform into capability-focused packages with clearly defined responsibilities and dependency rules, the architecture supports maintainability, extensibility and AI-assisted development.

All implementation SHALL conform to the package architecture defined within this document.
