# 1. Purpose

This document defines the Prompt Management Framework (PMF) for the 86-vibe platform.

The Prompt Management Framework provides a governed, version-controlled mechanism for creating, storing, validating, rendering, and executing prompts used throughout the platform.

Prompts are treated as first-class engineering artifacts and SHALL be managed with the same discipline as source code and architecture documentation.

This architecture ensures that AI-assisted development is deterministic, reproducible, auditable, and vendor-neutral.

---

# 2. Scope

This specification defines:

- Prompt architecture
- Prompt lifecycle
- Prompt storage
- Prompt rendering
- Prompt variables
- Prompt templates
- Prompt validation
- Prompt execution
- Prompt versioning
- Prompt metadata
- Testing requirements

This document does not define individual prompts or prompt engineering techniques.

---

# 3. Design Principles

The Prompt Management Framework SHALL adhere to the following principles.

## 3.1 Prompts as Code

Prompt templates SHALL be treated as version-controlled source artifacts.

## 3.2 Deterministic Rendering

The same prompt template and input variables SHALL always produce identical rendered prompts.

## 3.3 Vendor Neutrality

Prompt templates SHALL remain independent of specific AI providers wherever practical.

## 3.4 Human Governed

Prompt templates SHALL be reviewed through the normal Git workflow.

## 3.5 Reusable

Prompt templates SHALL be modular and reusable across repositories.

---

# 4. Architectural Overview

The Prompt Management Framework SHALL isolate prompt management from AI execution.

```text
          CLI
           │
           ▼
 Prompt Management Framework
           │
    ┌──────┼─────────┐
    │      │         │
    ▼      ▼         ▼
 Templates Variables Metadata
           │
           ▼
 Rendered Prompt
           │
           ▼
 AI Provider Abstraction Layer
```

The Prompt Management Framework SHALL not communicate directly with AI providers.

---

# 5. Responsibilities

The Prompt Management Framework SHALL be responsible for:

- Prompt discovery
- Prompt validation
- Variable substitution
- Metadata management
- Prompt rendering
- Prompt version tracking
- Template loading
- Prompt diagnostics

It SHALL NOT:

- Execute AI requests
- Select AI providers
- Perform repository operations
- Modify architecture documents

---

# 6. Prompt Repository

Prompt templates SHALL be stored within the platform repository.

Recommended directory structure:

```text
prompts/
│
├── architecture/
├── implementation/
├── review/
├── documentation/
├── testing/
├── workflows/
└── shared/
```

Repositories MAY contain additional prompt collections.

---

# 7. Prompt Template Format

Prompt templates SHALL be written using Markdown.

Templates SHALL support variable substitution using double braces.

Example:

```text
Project Name: {{ project_name }}

Repository: {{ repository }}

Feature: {{ feature_name }}
```

The rendering syntax SHALL remain provider independent.

---

# 8. Prompt Metadata

Every prompt SHALL include metadata.

Minimum metadata SHALL include:

```yaml
id: implement-feature
title: Feature Implementation
version: 1.0
category: implementation
owner: architecture
description: Generate implementation for an approved feature.
```

Future metadata fields MAY be added without breaking compatibility.

---

# 9. Prompt Categories

The platform SHALL organize prompts into logical categories.

| Category | Purpose |
|----------|---------|
| Architecture | Architecture generation and review |
| Implementation | Code generation |
| Documentation | Documentation generation |
| Testing | Test generation |
| Review | Architecture and code review |
| Workflow | Multi-step engineering workflows |
| Shared | Reusable prompt fragments |

---

# 10. Variable Management

Prompt variables SHALL be supplied separately from templates.

Variable sources SHALL include:

- CLI arguments
- Configuration
- Repository metadata
- User input
- Runtime context

Templates SHALL never contain hardcoded secrets.

---

# 11. Prompt Rendering

Prompt rendering SHALL follow the sequence below.

```text
Load Template
       │
       ▼
Load Metadata
       │
       ▼
Validate Variables
       │
       ▼
Substitute Variables
       │
       ▼
Validate Output
       │
       ▼
Return Rendered Prompt
```

Rendering SHALL fail if required variables are missing.

---

# 12. Prompt Validation

Validation SHALL include:

- Template syntax
- Metadata validation
- Required variables
- Unknown variables
- Empty templates
- Duplicate identifiers
- Invalid categories

Invalid templates SHALL not be rendered.

---

# 13. Prompt Discovery

The framework SHALL automatically discover available prompts.

Discovery SHALL identify:

- Prompt identifier
- Category
- Version
- Description
- Required variables

Prompt discovery SHALL support CLI listing commands.

---

# 14. Prompt Versioning

Prompt templates SHALL be versioned independently of platform releases.

Example:

```yaml
version: 1.2
```

Version updates SHALL occur whenever prompt behavior changes significantly.

---

# 15. Prompt Execution Flow

Prompt execution SHALL follow the architecture below.

```text
CLI Command
      │
      ▼
Prompt Framework
      │
      ▼
Render Prompt
      │
      ▼
AI Provider Abstraction Layer
      │
      ▼
Selected Provider
```

Prompt rendering SHALL always occur before AI execution.

---

# 16. CLI Integration

The following commands SHALL integrate with the Prompt Management Framework.

```text
86vibe prompt list

86vibe prompt show

86vibe prompt validate

86vibe prompt render
```

Future commands MAY include:

```text
86vibe prompt search

86vibe prompt inspect

86vibe prompt export
```

---

# 17. Logging Requirements

The Logging Service SHALL record:

- Prompt discovery
- Template loading
- Validation failures
- Successful rendering
- Missing variables
- Rendering duration

Rendered prompt content SHALL NOT be logged unless explicitly enabled for debugging.

---

# 18. Security Requirements

Prompt templates SHALL NOT contain:

- API keys
- Passwords
- Tokens
- Certificates
- Sensitive customer information

Secrets SHALL always be supplied through secure runtime mechanisms.

The framework SHALL prevent accidental exposure of secret values during rendering.

---

# 19. Extensibility

Future prompt capabilities MAY include:

- Template inheritance
- Partial templates
- Prompt composition
- Conditional rendering
- Localization
- Multi-language prompts

These capabilities SHALL be implemented without breaking existing templates.

---

# 20. Performance Requirements

The Prompt Management Framework SHALL:

- Cache discovered templates
- Minimize filesystem access
- Render prompts efficiently
- Support large prompt libraries

Rendering performance SHALL not become a bottleneck during normal CLI operations.

---

# 21. Testing Requirements

Automated tests SHALL verify:

- Prompt discovery
- Metadata parsing
- Template parsing
- Variable substitution
- Missing variables
- Unknown variables
- Invalid metadata
- Duplicate identifiers
- Version parsing
- Rendering determinism

Tests SHALL include representative prompt templates from multiple categories.

---

# 22. Definition of Done

The Prompt Management Framework SHALL be considered complete when:

- Prompt discovery functions correctly
- Metadata validation succeeds
- Variable substitution is deterministic
- Rendering is provider independent
- Templates are version controlled
- CLI integration is complete
- Automated tests pass
- Documentation is complete
