# 1. Purpose

This document defines the authoritative dependency relationships between the architecture packages, implementation packages, runtime services, Python packages, and external technology components comprising the 86-vibe platform.

Its objectives are to:

- establish deterministic implementation order
- identify mandatory and optional dependencies
- prevent cyclic dependencies
- enable parallel development where appropriate
- provide implementation sequencing for both human engineers and AI development agents
- ensure long-term maintainability of the platform architecture

This document extends the approved architecture by documenting dependency relationships only.

It SHALL NOT introduce new architectural capabilities or modify approved architecture.

---

# 2. Scope

This specification applies to:

- architecture packages
- implementation packages
- Python packages
- runtime services
- configuration components
- AI integration components
- validation components
- testing infrastructure
- build infrastructure
- Docker environment
- developer tooling

---

# 3. Dependency Principles

## 3.1 Directed Dependencies

Dependencies SHALL be directed.

If Component A depends on Component B, Component B SHALL NOT depend upon Component A.

---

## 3.2 No Circular Dependencies

Circular dependencies are prohibited.

Any dependency cycle SHALL be considered an architectural defect.

---

## 3.3 Stable Dependency Direction

Dependencies SHALL always point toward lower-level foundational services.

Higher-level orchestration components SHALL depend upon infrastructure components rather than the reverse.

---

## 3.4 Dependency Inversion

Where practical, dependencies SHALL target public interfaces rather than concrete implementations.

This enables:

- testing
- future extension
- provider substitution
- implementation flexibility

---

## 3.5 Explicit Dependencies

Every implementation dependency SHALL be documented.

Implicit dependencies SHALL be avoided.

---

# 4. Dependency Classification

Dependencies are classified into the following categories.

| Classification | Description |
|---------------|-------------|
| Mandatory | Required for correct operation |
| Optional | Used only when enabled |
| Development | Required only during development |
| Testing | Required only for testing |
| Build | Required during packaging and release |

# 8. Runtime Service Dependencies

The following diagram represents the approved runtime dependency hierarchy.

```
CLI

 ├── Bootstrap Manager
 │
 ├── Configuration Service
 │
 ├── Logging Service
 │
 ├── Repository Service
 │
 ├── Prompt Manager
 │
 ├── AI Provider Layer
 │
 ├── MCP Client
 │
 ├── Validation Framework
 │
 └── Service Lifecycle Manager
```

The CLI coordinates platform operation.

Individual services SHALL NOT depend upon the CLI.

# 12. Configuration Dependencies

| Component | Depends On |
|-----------|------------|
| Configuration Loader | None |
| Environment Provider | Configuration Loader |
| File Provider | Configuration Loader |
| Runtime Configuration | Environment Provider, File Provider |

Configuration SHALL be initialized before any other runtime service.

---

# 13. Logging Dependencies

Logging SHALL depend only upon configuration.

No other platform service SHALL be required to initialize logging.

This ensures early diagnostic capability during bootstrap.

---

# 14. AI Layer Dependencies

```
Configuration
      │
      ▼
Logging
      │
      ▼
Provider Adapter
      │
      ▼
Prompt Manager
      │
      ▼
MCP Client
```

The dependency order SHALL be preserved.

---

# 15. Repository Dependencies

Repository services depend upon:

- Configuration
- Logging

Repository services SHALL NOT depend upon:

- AI providers
- MCP
- Prompt management

This separation preserves repository independence.

---

# 16. Validation Dependencies

Validation depends upon:

- Repository services
- Configuration

Validation SHALL remain independent of AI providers.

Validation SHALL produce deterministic results.

---

# 17. Build Dependencies

| Build Activity | Dependencies |
|---------------|--------------|
| Static Analysis | Source Code |
| Unit Testing | Build Environment |
| Package Creation | Source Code |
| Docker Build | Package Build |
| Release Packaging | Successful Validation |

Build activities SHALL execute in the above order.

---

# 18. Testing Dependencies

```
Unit Tests
      │
      ▼
Integration Tests
      │
      ▼
System Tests
      │
      ▼
Reference Implementation Validation
```

Each testing phase depends upon successful completion of the previous phase.

---

# 19. Development Dependency Order

Implementation SHALL proceed in the following order.

1. Core Utilities
2. Configuration
3. Logging
4. Repository Services
5. Prompt Management
6. AI Provider Layer
7. MCP Integration
8. Validation
9. CLI
10. Bootstrap
11. Lifecycle Management
12. Reference Implementation

This sequence minimizes dependency conflicts and supports incremental validation.

---

# 20. Parallel Development Opportunities

The following components MAY be implemented concurrently after foundational services are complete.

| Parallel Group | Components |
|---------------|------------|
| Group A | Repository Services, Prompt Manager |
| Group B | AI Provider Layer, Validation Framework |
| Group C | CLI, Testing Framework |
| Group D | Build Infrastructure, Docker Environment |

Parallel development SHALL respect all documented dependencies.

---

# 21. Dependency Verification

The implementation SHALL verify:

- no circular imports
- no cyclic service dependencies
- no prohibited package references
- no dependency inversion violations
- no undocumented runtime dependencies

Automated verification is RECOMMENDED within the continuous integration pipeline.

---

# 22. Change Management

New dependencies SHALL require:

- architecture review
- impact assessment
- documentation update
- approval through the established architecture governance process

Changes affecting architectural layering SHALL require an approved ADR.

---

# 23. Conformance Criteria

This specification is satisfied when:

- all runtime dependencies are documented
- package dependencies are deterministic
- circular dependencies do not exist
- implementation order is defined
- service dependencies match approved architecture
- dependency verification can be automated

This document is complete when:

- architecture package dependencies are documented
- runtime service dependencies are defined
- Python package relationships are established
- implementation sequencing is deterministic
- parallel development opportunities are identified
- dependency verification requirements are specified
- no unresolved dependency ambiguity remains
