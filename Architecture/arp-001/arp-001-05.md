# 1. Purpose

This document establishes the mandatory engineering standards that SHALL be followed during implementation of the 86-vibe platform.

Its objectives are to:

- ensure a consistent codebase
- improve maintainability
- support collaboration between human engineers and AI development agents
- reduce code review effort
- maximize readability
- encourage deterministic implementations
- establish measurable quality expectations

This document extends the approved architecture by defining engineering standards only.

It SHALL NOT modify architectural behavior.

---

# 2. Scope

These standards apply to:

- all Python source code
- CLI commands
- tests
- configuration files
- documentation
- Docker assets
- CI/CD workflows
- generated code
- AI-generated code
- pull requests
- future platform services

---

# 3. Engineering Principles

Implementation SHALL follow the principles defined in AEP-001-04 and extend them with the following engineering principles.

## 3.1 Readability First

Code is written primarily for humans.

Readability SHALL take precedence over cleverness.

---

## 3.2 Deterministic Behavior

Code SHALL produce predictable results.

Hidden side effects SHALL be avoided.

---

## 3.3 Small Components

Functions and classes SHOULD have a single responsibility.

---

## 3.4 Explicitness

Implicit behavior SHOULD be minimized.

Dependencies SHALL be visible.

Configuration SHALL be explicit.

---

## 3.5 Testability

All implementation SHALL be written to enable automated testing.

---

## 3.6 Simplicity

The simplest implementation satisfying the architecture SHOULD be preferred.

---

# 4. Python Version

The platform SHALL target the Python version defined by the approved Technology Strategy.

All source code SHALL remain compatible with that supported version.

Platform-specific language features SHALL NOT be used unless approved through an ADR.

---

# 5. Code Formatting

Formatting SHALL be automated.

The project SHALL use:

| Standard | Requirement |
|----------|-------------|
| Indentation | Four spaces |
| Tabs | Prohibited |
| Line Endings | LF |
| File Encoding | UTF-8 |
| Final Newline | Required |

Maximum line length:

```
100 characters
```

Exceptions SHOULD be rare and justified.

---

# 6. Naming Standards

## Modules

```
snake_case.py
```

Example

```
configuration.py
prompt_manager.py
repository_service.py
```

---

## Packages

```
snake_case
```

---

## Classes

```
PascalCase
```

---

## Functions

```
snake_case()
```

---

## Variables

```
snake_case
```

---

## Constants

```
UPPER_SNAKE_CASE
```

---

## Private Members

Private members SHALL begin with a single underscore.

Example

```
_internal_state
```

Double underscore name mangling SHOULD NOT be used unless required.

---

# 7. Type Hints

Type hints SHALL be used throughout the platform.

Public APIs SHALL include complete type annotations.

Example

```python
def load_configuration(path: Path) -> Configuration:
```

The use of `Any` SHOULD be minimized.

---

# 8. Docstrings

Public modules SHALL contain module docstrings.

Public classes SHALL contain class docstrings.

Public functions SHALL contain docstrings.

The project SHALL use Google-style docstrings.

Example

```python
def validate(document: Path) -> ValidationResult:
    """
    Validate a document.

    Args:
        document:
            Document to validate.

    Returns:
        ValidationResult.
    """
```

---

# 9. Imports

Imports SHALL be grouped in the following order.

1. Standard library
2. Third-party libraries
3. Platform packages

Example

```python
from pathlib import Path

import typer

from vibe.configuration import Configuration
```

Wildcard imports are prohibited.

Relative imports SHOULD be avoided outside package-local modules.

---

# 10. Error Handling

Errors SHALL be explicit.

Exceptions SHALL NOT be silently ignored.

The following is prohibited.

```python
except Exception:
    pass
```

Errors SHALL either:

- be handled
- be logged
- be propagated

---

# 11. Logging

Logging SHALL use the platform logging framework.

The Python `print()` function SHALL NOT be used for diagnostics.

Log messages SHALL:

- be meaningful
- include context
- avoid sensitive information
- support troubleshooting

---

# 12. Configuration Access

Configuration SHALL only be accessed through the approved configuration service.

Environment variables SHALL NOT be read directly outside the configuration layer.

---

# 13. Dependency Injection

Platform services SHOULD receive dependencies through constructors.

Example

```python
class RepositoryService:
    def __init__(
        self,
        config: ConfigurationProvider,
        logger: Logger,
    ):
        ...
```

Global mutable state SHALL be avoided.

---

# 14. File Organization

Each module SHOULD contain related functionality only.

Very large modules SHOULD be refactored.

Guideline:

| Component | Preferred Maximum |
|-----------|-------------------|
| Module | 500 lines |
| Class | 300 lines |
| Function | 50 lines |

These values are guidelines rather than strict limits.

---

# 15. Public APIs

Public interfaces SHALL remain stable.

Implementation details SHALL remain private.

Breaking changes SHALL require:

- architecture review
- version increment
- documentation update

---

# 16. Testing Standards

Every public service SHALL have automated tests.

Testing SHALL include:

- unit tests
- integration tests where appropriate
- regression tests

Bug fixes SHALL include regression tests.

---

# 17. Code Duplication

Duplicate logic SHOULD be eliminated.

Shared functionality SHALL be extracted into reusable components.

Copy-and-paste programming is prohibited.

---

# 18. Comments

Comments SHOULD explain **why**, not **what**.

Obvious comments SHOULD be removed.

Example of poor comment

```python
# Increment counter
counter += 1
```

Example of useful comment

```python
# Retry because the provider may return transient rate-limit errors.
```

---

# 19. Security Standards

Implementation SHALL:

- validate external input
- avoid hard-coded secrets
- sanitize file paths
- use parameterized database access
- validate configuration
- fail securely

Credentials SHALL NOT appear in source code.

---

# 20. Performance Standards

Performance optimization SHALL follow measurement.

Premature optimization SHOULD be avoided.

Implementation SHALL favor:

- correctness
- readability
- maintainability

before optimization.

---

# 21. AI-Generated Code

AI-generated code SHALL be treated as human-written code.

It SHALL:

- satisfy architecture
- satisfy coding standards
- include tests
- undergo code review
- pass automated validation

No code SHALL bypass review solely because it was AI generated.

---

# 22. Pull Request Standards

Every pull request SHALL include:

- implementation summary
- affected architecture references
- testing performed
- documentation updates
- reviewer approval

Large pull requests SHOULD be avoided.

---

# 23. Static Analysis

The engineering pipeline SHALL perform automated static analysis.

The pipeline SHOULD include:

- formatting validation
- linting
- type checking
- unit testing
- documentation validation

Build failures SHALL block merging.

---

# 24. Continuous Integration

Every commit SHALL trigger automated validation.

Validation SHALL include:

1. formatting
2. linting
3. type checking
4. tests
5. architecture validation
6. documentation validation

No implementation SHALL bypass CI validation.

---

# 25. Documentation Standards

Public classes and services SHALL be documented.

Examples SHALL be provided where appropriate.

Documentation SHALL be updated within the same change that modifies implementation.

---

# 26. Code Review Checklist

Reviewers SHALL verify:

- architecture compliance
- coding standards
- naming conventions
- test coverage
- documentation updates
- dependency correctness
- error handling
- logging
- security
- readability

---

# 27. AI Development Agent Guidance

AI coding agents SHALL:

- follow approved architecture
- preserve canonical terminology
- generate type hints
- generate tests
- avoid introducing undocumented dependencies
- maintain package boundaries
- produce deterministic implementations
- avoid architectural redesign

Where architectural ambiguity is encountered, implementation SHALL stop and request clarification rather than inventing behavior.

---

# 28. Engineering Quality Gates

Implementation SHALL satisfy the following quality gates before merge.

| Gate | Required |
|------|----------|
| Builds Successfully | Yes |
| Formatting Passes | Yes |
| Static Analysis Passes | Yes |
| Type Checking Passes | Yes |
| Unit Tests Pass | Yes |
| Integration Tests Pass | When Applicable |
| Documentation Updated | Yes |
| Architecture Conformance Verified | Yes |
| Code Review Approved | Yes |

---

# 29. Definition of Engineering Ready

A work item is considered ready for implementation when:

- architecture exists
- dependencies are identified
- public interfaces are defined
- acceptance criteria are documented
- implementation sequencing is known

---

# 30. Definition of Code Complete

Code is considered complete when:

- implementation satisfies architecture
- tests pass
- documentation is updated
- quality gates pass
- code review is approved
- Definition of Done (AEP-001-18) is satisfied

---

# 31. Governance

These standards are mandatory for all contributors.

Exceptions SHALL require approval through the architecture governance process.

Changes to these standards SHALL require:

- architecture review
- version update
- repository approval

Major changes that alter engineering philosophy or implementation expectations SHOULD be captured in an ADR.

---

# 32. Conformance Criteria

This specification is satisfied when:

- implementation follows consistent coding standards
- automated quality gates are enforced
- AI-generated code follows the same standards as human-written code
- code reviews verify compliance
- documentation remains synchronized with implementation

---

# 33. Exit Criteria

This document is complete when:

- engineering standards are fully defined
- implementation expectations are documented
- quality gates are established
- review requirements are specified
- AI implementation guidance is complete
- the Architecture Readiness Package (ARP-001) is complete
- the platform is authorized to proceed to ARP-002 – Service Interface Contracts
