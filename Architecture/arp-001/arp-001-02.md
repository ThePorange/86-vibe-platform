# 1. Purpose

This document defines the canonical terminology used throughout the 86-vibe platform architecture.

Its objectives are to:

- establish a single authoritative vocabulary
- eliminate ambiguity across documentation
- ensure consistent language for engineers, AI coding agents, reviewers, and future contributors
- prevent multiple names being used for the same architectural concept
- provide a stable glossary for future architecture packages and implementation documentation

This document introduces no new architecture.

It standardizes the language used to describe the approved architecture.

---

# 2. Scope

This document applies to:

- architecture documents
- Architecture Decision Records (ADRs)
- implementation documentation
- source code
- API documentation
- CLI documentation
- developer guides
- automated code generation
- AI prompts
- issue tracking
- pull requests
- release documentation

All future documentation SHALL conform to the terminology defined herein.

---

# 3. Terminology Principles

## 3.1 Single Meaning

Every defined term SHALL have exactly one architectural meaning.

A term SHALL NOT represent multiple concepts.

---

## 3.2 Single Name

Every architectural concept SHALL have one preferred name.

Aliases SHALL be avoided unless required for compatibility.

---

## 3.3 Stable Vocabulary

Canonical terms SHALL remain stable across releases.

Changes SHALL require an approved ADR.

---

## 3.4 Consistency

The same terminology SHALL be used by:

- architecture documents
- implementation
- test suites
- documentation
- AI prompts
- developer tooling

---

## 3.5 Case Sensitivity

Unless explicitly stated otherwise:

- document titles use Title Case
- Python modules use snake_case
- Python classes use PascalCase
- functions use snake_case
- constants use UPPER_SNAKE_CASE
- CLI commands use kebab-case
- configuration keys use dot notation

# 4. Platform Terminology

|----------------|------------|

| Architecture | The approved technical design governing the platform. |
| Architecture Package (AP) | A collection of related architecture specifications. |
| Architecture Engineering Proposal (AEP) | An individual architecture specification document. |

# 15. Documentation Terminology

| Canonical Term | Definition |
|----------------|------------|

# 16. Normative Language

The following terms SHALL be interpreted according to their normative meanings.

| Term | Meaning |
|------|---------|
| SHALL | Mandatory requirement. |
| SHALL NOT | Mandatory prohibition. |
| SHOULD | Strong recommendation. |
| SHOULD NOT | Strong recommendation against. |
| MAY | Optional behavior. |
| RECOMMENDED | Preferred approach unless justified otherwise. |

These meanings SHALL be used consistently throughout all architecture and implementation documentation.

# 17. Naming Standards

## 17.1 Documents

Architecture documents SHALL use the following format:

```
AEP-001-07 Platform Architecture
```

Architecture readiness documents SHALL use:

```
ARP-001-02 Canonical Terminology
```

---

## 17.2 Python Packages

Python packages SHALL use:

```
snake_case
```

Example:

```
ai_provider
repository_service
prompt_manager
```

---

## 17.3 Python Classes

Python classes SHALL use:

```
PascalCase
```

Example:

```
PromptManager
ConfigurationProvider
RepositoryService
```

---

## 17.4 Functions

Functions SHALL use:

```
snake_case
```

Example:

```
load_configuration()
validate_document()
start_platform()
```

---

## 17.5 Constants

Constants SHALL use:

```
UPPER_SNAKE_CASE
```

Example:

```
DEFAULT_TIMEOUT
MAX_RETRY_COUNT
```

---

## 17.6 CLI Commands

CLI commands SHALL use:

```
kebab-case
```

Example:

```
86vibe init
86vibe validate
86vibe build
```

---

## 17.7 Configuration Keys

Configuration keys SHALL use dot notation.

Examples:

```
logging.level
provider.default
repository.root
mcp.timeout
```

---

# 18. Prohibited Terminology

The following practices SHALL NOT be used.

| Prohibited Practice | Reason |
|---------------------|--------|
| Multiple names for the same service | Creates ambiguity |
| Abbreviations without definition | Reduces readability |
| Vendor-specific terminology for generic concepts | Reduces portability |
| Informal aliases in architecture documents | Weakens consistency |
| Mixed naming conventions | Reduces implementation consistency |

---

# 19. AI Agent Guidance

AI implementation agents SHALL:

- use canonical names exactly as defined
- avoid inventing new terminology
- preserve capitalization
- preserve document identifiers
- generate code and documentation consistent with this specification
- treat this glossary as the authoritative vocabulary

---

# 20. Governance

Changes to canonical terminology SHALL require one of:

- an approved ADR
- a superseding version of this document

No implementation SHALL redefine terminology independently.

---

# 21. Conformance Criteria

This specification is satisfied when:

- all architecture documents use canonical terminology
- implementation uses the defined names
- generated documentation uses consistent vocabulary
- AI-generated artifacts conform to this glossary
- no conflicting terminology exists within the repository

---

# 22. Exit Criteria

This document is complete when:

- all major architectural concepts are defined
- normative language is standardized
- naming conventions are fully specified
- implementation terminology is unambiguous
- future documentation has a single authoritative glossary
