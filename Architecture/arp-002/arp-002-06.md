# 1. Purpose

This document defines the public implementation contract for the Prompt Management Service.

The Prompt Management Service is the authoritative platform component responsible for the discovery, loading, validation, rendering, versioning, and lifecycle management of prompt templates used throughout the 86-vibe platform.

Its objective is to ensure that all AI interactions use approved, deterministic prompt templates through a single public interface.

This document extends the approved architecture.

It SHALL NOT modify the architectural behaviour defined by:

- AEP-002-09 Prompt Management Framework Specification
- AEP-002-07 AI Provider Abstraction Layer Specification
- AEP-002-10 Repository Services Specification
- AEP-002-16 Platform Bootstrap Sequence Specification
- AEP-002-17 Service Lifecycle & Dependency Management Specification
- AEP-002-18 Platform Reference Implementation Specification

---

# 2. Scope

This specification defines:

- prompt management responsibilities
- prompt discovery
- template retrieval
- template rendering
- parameter validation
- prompt version management
- dependency contracts
- lifecycle behaviour
- extension requirements
- testing requirements

Implementation details are intentionally excluded.

---

# 3. Service Overview

The Prompt Management Service provides centralized management of all prompt templates used by the platform.

Its responsibilities include:

- discovering prompt templates
- loading prompt definitions
- validating prompt metadata
- validating template parameters
- rendering prompts
- exposing prompt metadata
- managing prompt versions
- caching prompt definitions where appropriate

The service SHALL be the exclusive mechanism through which prompt templates are accessed.

# 5. Responsibilities

The Prompt Management Service SHALL:

- discover approved prompt templates
- load prompt definitions
- validate prompt metadata
- render prompt templates
- validate supplied parameters
- expose prompt metadata
- provide prompt version information
- report prompt validation failures

The service SHALL NOT:

- execute AI inference
- select AI providers
- communicate with MCP servers
- access configuration directly
- implement business workflows

---

# 6. Public Interface

The service SHALL expose operations equivalent to the following logical interface.

```
initialize()

list_prompts()

get_prompt(identifier)

render(request)

validate(prompt)

reload()

health()

shutdown()
```

Equivalent language-specific implementations are permitted provided externally observable behaviour remains identical.

---

# 7. Lifecycle Contract

Initialization SHALL occur after:

- Configuration Service
- Logging Service
- Repository Service

Initialization sequence:

1. Construct service
2. Discover prompt repositories
3. Load prompt definitions
4. Validate prompt metadata
5. Build prompt index
6. Publish service availability

Shutdown SHALL release cached resources.

---

# 8. Dependency Contract

The Prompt Management Service SHALL depend upon:

- Configuration Service
- Logging Service
- Repository Service

The Prompt Management Service SHALL NOT depend upon:

- AI Provider Service
- MCP Client Service
- Validation Framework
- CLI

This dependency ordering preserves the approved architectural layering.

---

# 9. Prompt Discovery Contract

Prompt discovery SHALL identify approved prompt definitions available to the platform.

Each prompt SHALL expose metadata including:

- unique identifier
- display name
- description
- version
- parameter definitions
- template location
- status

Duplicate prompt identifiers SHALL be rejected.

---

# 10. Prompt Retrieval Contract

Prompt retrieval SHALL occur using a unique prompt identifier.

Consumers SHALL receive an immutable prompt definition.

Prompt retrieval SHALL be deterministic.

---

# 11. Prompt Rendering Contract

Prompt rendering SHALL accept logical information equivalent to:

- prompt identifier
- parameter values
- rendering options

Rendering SHALL:

- validate required parameters
- substitute parameter values
- preserve template formatting
- return the fully rendered prompt

Rendering SHALL NOT modify the stored template.

---

# 12. Parameter Validation Contract

The service SHALL validate:

- required parameters
- parameter names
- parameter types
- missing values
- duplicate parameters
- unsupported parameters

Validation SHALL occur before rendering.

---

# 13. Version Contract

Each prompt SHALL have a defined version.

The service SHALL expose prompt version metadata.

Consumers MAY request specific versions where supported by future architecture.

The default behaviour SHALL return the approved current version.

---

# 14. Repository Contract

Prompt templates SHALL be obtained exclusively through the Repository Service.

The Prompt Management Service SHALL NOT access repository storage directly.

Repository implementation details SHALL remain encapsulated.

---

# 15. Caching Contract

The service MAY cache validated prompt definitions.

Caching SHALL:

- preserve correctness
- remain transparent to consumers
- invalidate appropriately during reload

Consumers SHALL observe identical behaviour regardless of caching.

---

# 16. Error Contract

Errors SHALL be classified consistently.

| Error Type | Description |
|------------|-------------|
| Prompt Not Found | Requested prompt does not exist |
| Invalid Parameters | Required parameters missing or invalid |
| Template Error | Prompt template is malformed |
| Validation Failure | Prompt fails validation |
| Repository Failure | Prompt repository unavailable |
| Rendering Failure | Prompt could not be rendered |

Errors SHALL avoid exposing internal implementation details.

---

# 17. Logging Contract

The Prompt Management Service SHALL:

- log initialization
- log prompt discovery
- log validation failures
- log reload operations
- log rendering failures

Rendered prompt contents SHOULD NOT be logged unless explicitly enabled for diagnostic purposes.

Sensitive parameter values SHALL be masked.

---

# 18. Security Requirements

The service SHALL:

- validate template integrity
- validate parameter values
- prevent template injection where applicable
- protect confidential prompt content
- fail securely

Prompt templates SHALL be treated as trusted architecture-controlled assets.

---

# 19. Performance Expectations

The Prompt Management Service SHALL:

- optimize prompt retrieval
- minimize repeated template parsing
- support concurrent rendering
- avoid unnecessary repository access

Performance optimizations SHALL preserve deterministic rendering.

---

# 20. Thread Safety

The service SHALL support concurrent access.

Concurrent rendering operations SHALL:

- remain independent
- avoid shared mutable state
- preserve template integrity

Consumers SHALL NOT require explicit synchronization.

---

# 21. Extension Contract

Future prompt providers SHALL:

- implement the public prompt provider interface
- support prompt validation
- expose prompt metadata
- preserve rendering semantics

Existing consumers SHALL remain unaffected.

---

# 22. Consumer Contract

Consumers SHALL:

- obtain prompts only through the Prompt Management Service
- avoid loading templates directly
- treat prompt definitions as immutable
- provide validated parameter values

Consumers SHALL NOT modify prompt definitions.

---

# 23. Testing Requirements

Testing SHALL verify:

- initialization
- prompt discovery
- prompt retrieval
- parameter validation
- successful rendering
- rendering failures
- duplicate identifiers
- concurrent rendering
- reload behaviour
- graceful shutdown

Representative prompt templates SHOULD be used for deterministic testing.

---

# 24. AI Implementation Guidance

AI implementation agents SHALL:

- preserve prompt abstraction
- avoid embedding prompt text within business logic
- implement rendering through the public service interface
- preserve deterministic parameter substitution
- avoid introducing provider-specific prompt handling into this service

Prompt execution SHALL remain the responsibility of higher-level services.

---

# 25. Backward Compatibility

Future releases SHALL preserve:

- public service interface
- prompt identifiers
- rendering semantics
- parameter validation behaviour
- version reporting

Breaking changes SHALL require:

- architecture review
- semantic version increment
- approved ADR where applicable

---

# 26. Conformance Criteria

This service contract is satisfied when:

- prompt templates are accessed exclusively through the Prompt Management Service
- prompt rendering is deterministic
- parameter validation conforms to this specification
- repository access remains encapsulated
- lifecycle behaviour conforms to this specification
- concurrency requirements are satisfied

- dependency contracts are established
- rendering and validation contracts are documented
- extension requirements are defined
