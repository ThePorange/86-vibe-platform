# 1. Purpose

This document defines the public implementation contract for the Platform Command Line Interface (CLI).

The purpose of this contract is to ensure that every implementation of the CLI behaves consistently regardless of implementation details.

This specification SHALL be treated as the authoritative interface definition for:

- human engineers
- Cursor
- Claude Code
- OpenAI Codex
- Gemini CLI
- future implementation teams

This document extends the approved architecture.

It SHALL NOT modify architectural behaviour defined in:

- AEP-002-04 CLI Specification
- AEP-002-16 Platform Bootstrap Sequence
- AEP-002-17 Service Lifecycle
- AEP-002-18 Reference Implementation

---

# 2. Scope

This contract specifies:

- public CLI behaviour
- supported commands
- command execution lifecycle
- exit codes
- input validation
- output requirements
- dependency contracts
- error contracts
- extension requirements

It intentionally excludes implementation details.

---

# 3. Service Overview

The CLI is the primary entry point into the platform.

Responsibilities include:

- bootstrap platform execution
- initialize runtime services
- dispatch user commands
- provide consistent user feedback
- return deterministic exit codes
- expose administrative operations
- expose developer tooling

The CLI SHALL remain thin.

Business logic SHALL reside in platform services.

# 5. Public Interface

The CLI SHALL expose a single executable entry point.

```
86vibe
```

The executable SHALL support subcommands.

---

# 6. Required Commands

The following commands SHALL exist.

| Command | Purpose |
|----------|----------|
| init | Initialize a workspace |
| validate | Validate repository artifacts |
| build | Build platform artifacts |
| config | Manage configuration |
| providers | Display AI providers |
| prompts | Manage prompts |
| repository | Repository operations |
| mcp | MCP operations |
| doctor | Diagnostic validation |
| version | Display version information |
| help | Display help |

Additional commands MAY be added in future releases.

---

# 7. Command Structure

Commands SHALL use the following format.

```
86vibe <command> [options]
```

Nested commands SHALL use:

```
86vibe repository status
```

Example:

```
86vibe prompts list
```

---

# 8. Startup Contract

Upon execution the CLI SHALL perform the following sequence.

1. Parse arguments
2. Validate command
3. Initialize bootstrap manager
4. Load configuration
5. Initialize logging
6. Initialize required services
7. Execute command
8. Return exit code

The sequence SHALL be deterministic.

---

# 9. Dependency Contract

The CLI SHALL depend upon:

- Bootstrap Service
- Configuration Service
- Logging Service
- Lifecycle Manager

The CLI SHALL NOT directly depend upon:

- provider implementations
- repository implementations
- prompt implementations

Interaction SHALL occur through public service interfaces.

---

# 10. Input Contract

The CLI SHALL accept:

- command line arguments
- configuration
- environment variables
- standard input where appropriate

Inputs SHALL be validated before execution.

---

# 11. Output Contract

Successful commands SHALL provide:

- human-readable output
- deterministic formatting
- machine-readable output where supported

Output SHALL be:

- consistent
- concise
- informative

---

# 12. Exit Codes

The CLI SHALL return the following exit codes.

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | General Error |
| 2 | Invalid Arguments |
| 3 | Configuration Error |
| 4 | Validation Failure |
| 5 | Runtime Failure |
| 6 | External Service Failure |
| 7 | Internal Platform Error |

Exit codes SHALL remain stable.

---

# 13. Error Contract

Errors SHALL:

- produce meaningful messages
- be logged
- return deterministic exit codes
- avoid stack traces unless diagnostic mode is enabled

Example

```
Configuration file not found.
```

NOT

```
KeyError at line 142...
```

---

# 14. Help Contract

Every command SHALL support:

```
--help
```

Help output SHALL include:

- syntax
- options
- examples
- description

---

# 15. Version Contract

The CLI SHALL support:

```
86vibe version
```

Output SHALL include:

- platform version
- architecture version
- Python version
- build identifier

---

# 16. Logging Contract

The CLI SHALL:

- initialize logging before command execution
- respect configured log levels
- avoid direct console logging outside the logging framework

---

# 17. Configuration Contract

The CLI SHALL NOT read configuration files directly.

Configuration SHALL be obtained exclusively through the Configuration Service.

---

# 18. Thread Safety

Multiple CLI invocations SHALL execute independently.

The CLI SHALL NOT rely upon mutable shared process state.

---

# 19. Extension Contract

New commands SHALL:

- follow existing naming conventions
- use dependency injection
- support help output
- return defined exit codes
- participate in bootstrap initialization

---

# 20. Validation Requirements

The CLI SHALL validate:

- command existence
- option syntax
- required parameters
- configuration availability
- service readiness

before executing command logic.

---

# 21. Performance Expectations

Startup SHOULD complete within one second under normal development environments.

Individual command performance SHALL be governed by the service being invoked.

---

# 22. Security Requirements

The CLI SHALL:

- validate file paths
- sanitize user input
- avoid exposing secrets
- respect filesystem permissions
- fail securely

---

# 23. Testing Requirements

Testing SHALL include:

- command parsing
- invalid arguments
- exit codes
- help output
- version output
- bootstrap failures
- configuration failures
- service failures

---

# 24. AI Implementation Guidance

AI implementation agents SHALL:

- implement only documented commands
- preserve exit codes
- preserve command syntax
- avoid embedding business logic in the CLI
- delegate functionality to platform services

---

# 25. Backward Compatibility

Future releases SHALL preserve:

- command names
- option names
- exit codes

Breaking changes SHALL require:

- architecture review
- semantic version increment
- approved ADR where applicable

---

# 26. Conformance Criteria

This service contract is satisfied when:

- all required commands exist
- startup sequence conforms
- dependencies conform
- output is deterministic
- exit codes match this specification
- testing verifies public behaviour

---

# 27. Exit Criteria

This document is complete when:

- the CLI public interface is fully specified
- lifecycle expectations are defined
- dependency contracts are established
- error handling is standardized
- extension rules are documented
- implementation teams can develop the CLI without further architectural clarification
