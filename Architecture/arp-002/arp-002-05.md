# 1. Purpose

This document defines the public implementation contract for the MCP Client Service.

The MCP Client Service is responsible for providing a standardized interface between the 86-vibe platform and external Model Context Protocol (MCP) servers.

Its purpose is to isolate the remainder of the platform from protocol-specific implementation details while providing deterministic discovery, connection, invocation, and lifecycle management of MCP resources.

This document extends the approved architecture.

It SHALL NOT modify the architectural behaviour defined by:

- AEP-002-08 Model Context Protocol (MCP) Integration Specification
- AEP-002-07 AI Provider Abstraction Layer Specification
- AEP-002-16 Platform Bootstrap Sequence Specification
- AEP-002-17 Service Lifecycle & Dependency Management Specification
- AEP-002-18 Platform Reference Implementation Specification

# 2. Scope

This specification defines:

- MCP client responsibilities
- server discovery
- server registration
- connection lifecycle
- tool invocation
- resource access
- prompt retrieval
- capability discovery
- dependency contracts
- extension requirements
- testing requirements

Implementation details are intentionally excluded.

---

# 3. Service Overview

The MCP Client Service provides a protocol-independent interface for interacting with one or more MCP servers.

Its responsibilities include:

- discovering configured MCP servers
- establishing server connections
- negotiating protocol capabilities
- invoking tools
- accessing resources
- retrieving prompts
- managing connection lifecycle
- reporting server health

The service SHALL hide protocol implementation details from consumers.

# 5. Responsibilities

The MCP Client Service SHALL:

- manage MCP server registrations
- establish client connections
- expose server capabilities
- invoke remote tools
- retrieve remote resources
- retrieve server-defined prompts
- normalize protocol responses
- manage connection lifecycle

The service SHALL NOT:

- execute business workflows
- construct AI prompts
- communicate directly with AI models
- expose protocol implementation details
- bypass platform configuration

---

# 6. Public Interface

The service SHALL expose operations equivalent to the following logical interface.

```
initialize()

list_servers()

connect(server)

disconnect(server)

list_tools(server)

list_resources(server)

list_prompts(server)

invoke_tool(request)

get_resource(request)

get_prompt(request)

health()

shutdown()
```

Equivalent language-specific implementations are permitted provided observable behaviour remains identical.

---

# 7. Lifecycle Contract

Initialization SHALL occur after:

- Configuration Service
- Logging Service
- AI Provider Service

Initialization sequence:

1. Construct service
2. Load MCP configuration
3. Discover configured servers
4. Register servers
5. Validate server definitions
6. Establish configured connections
7. Publish service availability

Shutdown SHALL gracefully close all active connections.

---

# 8. Dependency Contract

The MCP Client Service SHALL depend upon:

- Configuration Service
- Logging Service
- AI Provider Service

The MCP Client Service SHALL NOT depend upon:

- Prompt Manager
- Repository Service
- Validation Framework
- CLI

This dependency ordering preserves the approved architecture and prevents circular dependencies.

---

# 9. Server Registration Contract

Each configured MCP server SHALL expose:

- unique identifier
- display name
- transport configuration
- operational status
- supported protocol version
- advertised capabilities

Duplicate server identifiers SHALL be rejected.

---

# 10. Connection Contract

Each server connection SHALL support:

- initialization
- capability negotiation
- health verification
- graceful shutdown
- automatic cleanup

Connection state SHALL be tracked internally.

---

# 11. Capability Discovery Contract

The service SHALL expose standardized discovery operations for:

- tools
- resources
- prompts

Capability information SHALL be normalized before presentation to consumers.

---

# 12. Tool Invocation Contract

Tool invocation SHALL include logical request information equivalent to:

- server identifier
- tool identifier
- input parameters
- execution options

Invocation SHALL return normalized results independent of server implementation.

---

# 13. Resource Access Contract

Resources SHALL be retrieved through the MCP Client Service.

Consumers SHALL NOT communicate directly with MCP servers.

Retrieved resources SHALL preserve protocol semantics while exposing a provider-independent interface.

---

# 14. Prompt Retrieval Contract

The service SHALL support retrieval of prompts exposed by MCP servers.

Prompt metadata SHALL include information equivalent to:

- prompt identifier
- display name
- description
- parameter definitions
- availability status

Prompt execution remains the responsibility of higher-level platform services.

---

# 15. Health Contract

The service SHALL provide health information for:

- individual servers
- active connections
- protocol negotiation
- transport availability

Health information SHALL be available without exposing implementation-specific diagnostics.

---

# 16. Error Contract

Errors SHALL be classified consistently.

| Error Type | Description |
|------------|-------------|
| Server Not Found | Requested server unavailable |
| Connection Failure | Unable to establish connection |
| Capability Not Found | Requested capability unavailable |
| Invocation Failure | Tool execution failed |
| Resource Failure | Resource unavailable |
| Protocol Error | MCP protocol violation |
| Timeout | Server failed to respond |
| Authentication Failure | Server authentication failed |

Errors SHALL be normalized before presentation to consumers.

---

# 17. Logging Contract

The MCP Client Service SHALL:

- log server registration
- log connection establishment
- log capability discovery
- log tool invocation
- log failures
- log connection shutdown

Sensitive request data SHALL NOT be logged unless explicitly enabled through configuration.

---

# 18. Security Requirements

The service SHALL:

- validate server configuration
- validate server identity where supported
- protect authentication credentials
- sanitize remote errors
- reject malformed protocol messages
- fail securely

Authentication credentials SHALL remain internal to the service.

---

# 19. Performance Expectations

The MCP Client Service SHALL:

- reuse established connections where practical
- minimize connection overhead
- avoid repeated capability discovery unless required
- support concurrent requests to multiple servers

Performance optimizations SHALL preserve deterministic behaviour.

---

# 20. Thread Safety

The service SHALL support concurrent access.

Concurrent requests SHALL:

- maintain connection integrity
- avoid resource corruption
- avoid deadlocks

Consumers SHALL NOT require explicit synchronization.

---

# 21. Extension Contract

Future MCP transport implementations SHALL:

- implement the common client interface
- support standardized lifecycle operations
- support capability discovery
- preserve normalized responses

Transport-specific behaviour SHALL remain internal.

---

# 22. Consumer Contract

Consumers SHALL:

- communicate only through the MCP Client Service
- avoid direct protocol communication
- treat server implementations as opaque
- avoid transport-specific behaviour

The service SHALL remain the sole integration point for MCP.

---

# 23. Testing Requirements

Testing SHALL verify:

- service initialization
- server registration
- connection establishment
- capability discovery
- tool invocation
- resource retrieval
- prompt retrieval
- connection failures
- protocol failures
- concurrent access
- graceful shutdown

Mock MCP servers SHOULD be used for deterministic testing.

---

# 24. AI Implementation Guidance

AI implementation agents SHALL:

- implement transport adapters behind the public interface
- normalize protocol responses
- preserve protocol abstraction
- avoid exposing transport-specific APIs
- maintain deterministic connection lifecycle behaviour

Protocol implementation details SHALL remain encapsulated.

---

# 25. Backward Compatibility

Future releases SHALL preserve:

- public service interface
- lifecycle semantics
- capability discovery behaviour
- normalized request and response contracts
- error classifications

Breaking changes SHALL require:

- architecture review
- semantic version increment
- approved ADR where applicable

# 26. Conformance Criteria

This service contract is satisfied when:

- consumers interact exclusively through the MCP Client Service
- protocol implementation remains encapsulated
- capability discovery is standardized
- tool invocation is normalized
- connection lifecycle conforms to this specification
- concurrency requirements are satisfied
