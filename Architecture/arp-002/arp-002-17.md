# 4. Service Definition
The Platform Context Service is the authoritative runtime service for creating, obtaining, propagating, cloning, and releasing immutable execution contexts.
An execution context represents one logical unit of platform work.
The service SHALL provide:
- stable execution identity
- correlation identity
- request identity
- originating-service identity
- platform metadata
- execution metadata
- deterministic context propagation
- context isolation
- lifecycle-controlled resource cleanup
The Platform Context Service SHALL NOT act as a general-purpose mutable data store.
---
# 5. Service Responsibilities
The Platform Context Service SHALL:
1. Construct the immutable platform context during service initialization.
2. Create immutable execution contexts for logical operations.
3. Generate context identifiers when callers do not supply approved identifiers.
4. Generate or preserve correlation identifiers according to this specification.
5. Associate a current execution context with the active execution scope.
6. Return the current execution context when one is associated.
7. Clone contexts according to the approved clone semantics.
8. Propagate execution context across supported synchronous and asynchronous boundaries.
9. Release execution-scope associations and service-managed context resources.
10. Validate all caller-supplied context creation data.
11. Prevent cross-request mutable state.
12. Expose service health through the approved lifecycle and health contracts.
13. Preserve deterministic behavior across repeated equivalent operations.
14. Integrate with the Configuration Service, Logging Service, and Service Registry.
15. Protect context metadata from unauthorized mutation and disclosure.
---
# 6. Explicit Non-Responsibilities
The Platform Context Service SHALL NOT:
- execute application or business workflows
- schedule work
- manage threads, processes, or event loops
- initialize or shut down unrelated services
- act as the Service Registry
- act as the Service Lifecycle Manager
- persist execution contexts to a database
- persist execution contexts to repository storage
- communicate with AI providers
- communicate with MCP servers
- render prompts
- validate repository artifacts
- provide distributed tracing transport
- provide authentication or authorization
- store credentials or secrets
- provide arbitrary global mutable state
- automatically serialize contexts across process boundaries
Cross-process context transport MAY be introduced only through future approved architecture.
---
# 7. Service Capabilities
The service SHALL provide the following capabilities.
| Capability | Description |
|------------|-------------|
| Platform Context Construction | Creates immutable process-wide platform metadata |
| Execution Context Creation | Creates a new context for one logical operation |
| Current Context Resolution | Resolves the context associated with the active execution scope |
| Explicit Context Binding | Associates a context with the active execution scope |
| Context Propagation | Preserves context across supported execution boundaries |
| Context Cloning | Produces a new immutable context derived from an existing context |
| Context Release | Removes execution-scope associations and service-owned resources |
| Context Validation | Validates identifiers and metadata |
| Context Health Reporting | Reports service readiness and operational health |
| Context Inspection | Exposes immutable context data through approved models |
The implementation SHALL NOT add capabilities outside this list without architecture approval.

# 8. Public Service Interface
The Platform Context Service SHALL expose operations equivalent to:
```text
initialize()
create(request)
current(required = false)
bind(context)
clone(context, request = null)
release(context = null)
platform()
health()
shutdown()

ARP-002-15 remains authoritative for the existence of:

initialize()
create(request)
current()
platform()
clone(context)
release(context)
health()
shutdown()

This addendum clarifies:

* current() accepts logical required/optional behavior.
* bind(context) is the explicit context-association operation needed to support deterministic current-context resolution.
* clone(context, request) MAY receive clone overrides.
* release(context) MAY release an explicit context or the current context.

Equivalent language-specific signatures MAY be used provided all behavior defined herein is preserved.

9. Initialize Operation Contract

9.1 Purpose

initialize() prepares the service for context creation and resolution.

9.2 Preconditions

The following services SHALL be available:

* Configuration Service
* Logging Service
* Service Registry

9.3 Behavior

Initialization SHALL:

1. Verify that the service is not already initialized.
2. Load and validate Platform Context Service configuration.
3. Construct immutable platform metadata.
4. Initialize identifier generation.
5. Initialize execution-scope storage.
6. Register the service through the Service Registry.
7. Publish service readiness.
8. Log successful initialization.

9.4 Idempotency

Repeated initialization after successful initialization SHALL:

* return the established initialized state; or
* produce a documented already-initialized result.

It SHALL NOT construct a second independent service state.

9.5 Failure

Initialization failure SHALL leave the service unavailable.

Partially initialized resources SHALL be released before the failure is returned.

⸻

10. Create Operation Contract

10.1 Purpose

create(request) creates a new immutable execution context.

10.2 Request Model

The logical creation request SHALL support:

Field	Required	Description
correlation_id	No	Existing correlation identity to preserve
request_id	No	Caller-supplied request identity
originating_service	Yes	Canonical service identifier
request_metadata	No	Approved request metadata
execution_metadata	No	Approved execution metadata
bind	No	Whether the created context becomes current
parent_context_id	No	Parent identity for derived work

10.3 Identifier Behavior

When absent:

* context_id SHALL always be generated by the service.
* correlation_id SHALL be generated by the service.
* request_id MAY be generated by the service when required by the caller-facing workflow.

Caller-supplied context_id SHALL NOT be accepted.

10.4 Creation Behavior

The service SHALL:

1. Verify readiness.
2. Validate the request.
3. Generate a unique context identifier.
4. Preserve or generate the correlation identifier.
5. Validate metadata limits and reserved keys.
6. combine immutable platform metadata references with request and execution metadata.
7. Set the creation timestamp.
8. Construct an immutable context.
9. Bind the context when requested.
10. Return the context.

10.5 Determinism

Equivalent supplied metadata SHALL produce semantically equivalent contexts, excluding service-generated identifiers and timestamps.

⸻

11. Current Operation Contract

11.1 Purpose

current(required = false) returns the context associated with the active execution scope.

11.2 Optional Resolution

When no context is associated and required is false:

* the operation SHALL return no context; and
* it SHALL NOT create an implicit context.

11.3 Required Resolution

When no context is associated and required is true:

* the operation SHALL raise or return ContextNotFound.

11.4 Isolation

The current context SHALL be isolated between:

* independent logical requests
* concurrent threads
* concurrent asynchronous tasks

A context bound in one execution scope SHALL NOT become current in an unrelated execution scope.

11.5 No Hidden Fallback

The platform context SHALL NOT be returned as a substitute for a missing execution context.

⸻

12. Bind Operation Contract

12.1 Purpose

bind(context) associates an existing valid execution context with the active execution scope.

12.2 Validation

The service SHALL verify:

* the context conforms to the approved model
* the context has not been released
* the context belongs to the current service instance where ownership is tracked
* the context is immutable
* required identifiers are present and valid

12.3 Behavior

Binding SHALL return a binding handle or equivalent lifecycle token when required by the implementation language.

The binding mechanism SHALL support deterministic restoration of any previously bound context.

12.4 Nesting

Nested bindings MAY be supported.

When nested binding is supported:

* bindings SHALL behave as a stack
* releasing the nested binding SHALL restore the immediately preceding context
* unrelated execution scopes SHALL remain unaffected

⸻

13. Clone Operation Contract

13.1 Purpose

clone(context, request = null) creates a new immutable context derived from an existing context.

13.2 Clone Identity

A cloned context SHALL:

* receive a new context_id
* preserve the source correlation_id by default
* identify the source context as parent_context_id
* preserve platform metadata
* preserve originating service unless explicitly replaced by an approved override
* receive a new creation timestamp

13.3 Metadata

Clone metadata SHALL be constructed by:

1. copying source metadata
2. applying approved override metadata
3. rejecting reserved-key modification
4. validating the final metadata

13.4 Source Integrity

Cloning SHALL NOT modify the source context.

13.5 Binding

A clone SHALL become current only when explicitly requested.

⸻

14. Release Operation Contract

14.1 Purpose

release(context = null) releases a context association and any service-owned resources associated with that context.

14.2 Default Behavior

When no explicit context is supplied, the service SHALL target the context associated with the active execution scope.

14.3 Idempotency

Releasing an already released or unbound context SHALL be idempotent when ownership can be established safely.

It SHALL NOT corrupt another context association.

14.4 Post-Release Behavior

After release:

* the released context SHALL NOT be returned by current()
* the context SHALL NOT be rebound if the implementation tracks it as permanently released
* service-owned binding handles SHALL be invalidated
* references held by external consumers remain immutable data but SHALL no longer represent an active execution scope

14.5 Nested Binding

Where nested binding is used, release SHALL restore the previous valid binding.

⸻

15. Platform Operation Contract

platform() SHALL return the immutable process-wide platform context.

The platform context SHALL include:

* platform identifier
* application version
* architecture baseline identifier
* runtime environment
* active configuration profile
* process start timestamp where approved
* service instance identifier where approved

The platform context SHALL:

* be created once per service instance
* remain immutable
* remain available after successful initialization
* not contain request-specific data
* not contain credentials or secrets

⸻

16. Health Operation Contract

health() SHALL report:

* initialization status
* readiness status
* execution-scope storage availability
* identifier generator availability
* configuration validity
* registry integration status

Health evaluation SHALL NOT expose:

* context metadata
* request identifiers
* correlation identifiers
* secrets
* internal execution-scope contents

⸻

17. Internal Component Architecture

The Platform Context Service SHALL be decomposed into the following logical components.

Component	Responsibility
PlatformContextService	Public facade and lifecycle coordination
PlatformContextFactory	Constructs immutable platform context
ExecutionContextFactory	Creates and clones execution contexts
ContextValidator	Validates requests, contexts, identifiers, and metadata
ContextIdentifierProvider	Generates unique context and correlation identifiers
ContextScopeStore	Associates contexts with active execution scopes
ContextBinding	Represents a reversible context association
ContextMetadataPolicy	Enforces metadata limits and reserved-key rules
ContextHealthProvider	Reports service health
Context Models	Immutable request, context, metadata, status, and error models

17.1 Facade Rule

Consumers SHALL depend on PlatformContextService and public models.

Consumers SHALL NOT depend directly on internal factories, validators, stores, or policies.

17.2 Component Independence

Internal components SHALL communicate through explicit interfaces.

Global mutable dictionaries or module-level context variables SHALL NOT serve as the architectural service implementation.

17.3 Scope Storage

ContextScopeStore SHALL use execution-scope-local storage appropriate to the runtime language.

For Python, the implementation SHOULD use an asynchronous-context-safe mechanism rather than thread-local storage alone.

The storage mechanism SHALL preserve isolation across asynchronous tasks.

⸻

18. Configuration Architecture

18.1 Configuration Namespace

Configuration SHALL use the namespace:

context.*

18.2 Configuration Keys

The service SHALL support the following logical configuration.

Key	Type	Default	Required	Purpose
context.enabled	Boolean	true	No	Enables service initialization
context.bind_on_create	Boolean	false	No	Default binding behavior
context.require_originating_service	Boolean	true	No	Enforces source-service identity
context.max_request_metadata_entries	Integer	32	No	Maximum request metadata entries
context.max_execution_metadata_entries	Integer	32	No	Maximum execution metadata entries
context.max_metadata_key_length	Integer	128	No	Maximum metadata-key length
context.max_metadata_value_length	Integer	4096	No	Maximum scalar metadata-value length
context.allow_nested_bindings	Boolean	true	No	Controls nested execution scopes
context.log_lifecycle_events	Boolean	true	No	Controls context lifecycle diagnostics
context.log_identifiers	Boolean	false	No	Controls identifier logging
context.fail_on_release_mismatch	Boolean	true	No	Controls mismatched release behavior

18.3 Validation Rules

Configuration SHALL be rejected when:

* numeric limits are less than zero
* metadata key length is zero
* metadata value length is zero
* a value has an invalid type
* unsupported configuration keys are treated as errors by the Configuration Service policy
* configuration would disable a mandatory service required by the approved runtime profile

18.4 Configuration Access

The service SHALL obtain configuration exclusively from the Configuration Service.

The service SHALL NOT:

* read environment variables directly
* parse configuration files directly
* mutate configuration
* retain mutable configuration objects

18.5 Reload Behavior

Runtime configuration reload SHALL NOT alter existing contexts.

Configuration reload MAY affect subsequently created contexts only when:

* the Configuration Service supports reload
* the Platform Context Service explicitly validates the new configuration
* the reload does not alter immutable platform context fields

Changes requiring reconstruction of platform context SHALL require service restart.

⸻

19. Lifecycle Architecture

The service SHALL participate in the following states:

Uninitialized
    ↓
Initializing
    ↓
Ready
    ↓
Stopping
    ↓
Stopped

Failure MAY transition the service to:

Failed

19.1 Initialization Order

The service SHALL initialize after:

1. Configuration Service
2. Logging Service
3. Service Registry
4. Bootstrap foundation required by the approved startup sequence
5. Service Lifecycle Manager availability required for registration

19.2 Ready State

The service SHALL enter Ready only when:

* configuration is valid
* platform context exists
* identifier generation is available
* scope storage is available
* service registration succeeds
* health reporting is available

19.3 Shutdown

Shutdown SHALL:

1. reject new context creation
2. prevent new bindings
3. clear active service-owned scope associations
4. invalidate service-owned binding handles
5. unregister health integration where required
6. release internal resources
7. transition to Stopped

19.4 Shutdown Idempotency

Repeated shutdown SHALL not fail or recreate resources.

22. Error Model

The service SHALL expose canonical errors equivalent to:

Error	Meaning
ContextServiceNotInitialized	Operation attempted before readiness
ContextServiceStopped	Operation attempted after shutdown began
InvalidContextRequest	Creation or clone request invalid
InvalidContext	Context does not conform to the approved model
ContextNotFound	Required current or requested context unavailable
ContextAlreadyReleased	Operation attempted on a released context
ContextBindingConflict	Binding violates scope rules
ContextReleaseMismatch	Released context does not match active binding
ContextMetadataViolation	Metadata violates limits or reserved-key rules
ContextCreationFailure	Context could not be constructed
ContextConfigurationError	Context configuration invalid
ContextRegistrationError	Service Registry registration failed
ContextInternalError	Unexpected internal failure

22.1 Error Requirements

Errors SHALL:

* use canonical error types
* include a stable machine-readable error code
* include a safe human-readable message
* preserve the originating cause internally where supported
* avoid secrets and full context metadata
* avoid implementation-specific exception leakage

22.2 Programmer Errors

Mutation attempts against immutable context models SHOULD fail immediately.

⸻

23. Logging Requirements

The service SHALL use the Logging Service exclusively.

23.1 Required Events

The service SHALL log:

* initialization start
* initialization success
* initialization failure
* service registration failure
* configuration validation failure
* context creation failure
* binding conflict
* release mismatch
* propagation failure
* shutdown start
* shutdown completion
* unexpected internal failure

23.2 Optional Diagnostic Events

When enabled, the service MAY log:

* context created
* context bound
* context cloned
* context released

23.3 Identifier Logging

Context, request, and correlation identifiers SHALL be treated as operational metadata.

They SHALL only be logged when:

* configuration permits identifier logging; or
* an error requires safe correlation.

23.4 Prohibited Logging

Logs SHALL NOT contain:

* complete request metadata
* complete execution metadata
* credentials
* authentication tokens
* provider prompts or responses
* arbitrary user payloads
* secrets

⸻

24. Security Requirements

The service SHALL:

1. Treat all externally supplied metadata as untrusted.
2. Validate metadata types, sizes, keys, and values.
3. Reject reserved metadata-key modification.
4. Prevent callers from supplying service-generated context identifiers.
5. Prevent mutation of identifiers after creation.
6. Prevent context leakage between execution scopes.
7. Prevent secrets from being stored in context.
8. Avoid serializing context automatically.
9. Avoid logging full context content.
10. Clear service-owned context associations during shutdown.
11. Reject invalid or forged context models at binding boundaries.
12. Fail securely on configuration or validation errors.

24.1 Reserved Metadata

The following logical keys SHALL be reserved:

* context_id
* correlation_id
* request_id
* parent_context_id
* created_at
* platform_id
* platform_version
* architecture_version
* originating_service

Callers SHALL NOT override reserved metadata through generic metadata mappings.

⸻

25. Performance Requirements

The service SHALL meet the following architectural expectations under normal local operation.

Operation	Expectation
Current-context lookup	Constant-time
Context binding	Constant-time
Context release	Constant-time
Platform-context lookup	Constant-time
Context creation	Linear only with supplied metadata size
Context cloning	Linear only with copied metadata size

The service SHALL:

* avoid network access
* avoid filesystem access during routine operations
* avoid repository access
* avoid blocking I/O
* avoid unbounded metadata
* avoid retaining released contexts
* support concurrent execution contexts
* minimize object duplication while preserving immutability

No strict wall-clock service-level objective is established by this architecture.

Performance acceptance SHALL be based on algorithmic behavior, bounded resources, and absence of unnecessary I/O.

⸻

26. Resource Lifecycle

26.1 Owned Resources

The service owns:

* immutable platform context
* identifier provider
* execution-scope store
* active binding handles
* service configuration snapshot
* health provider
* service registration state

26.2 Context Ownership

An execution context is immutable value data.

The service owns:

* its creation
* its active scope association
* its binding lifecycle

Consumers MAY retain context references for propagation and inspection but SHALL NOT mutate them.

26.3 Release Rules

The service SHALL ensure:

* active scope associations are removable
* nested bindings restore prior state correctly
* released bindings are not reused
* shutdown clears service-owned scope associations
* released contexts are not retained in an unbounded service collection

26.4 No Persistence

Context state SHALL remain in process memory only.

The service SHALL NOT persist contexts.

26.5 Leak Prevention

The implementation architecture SHALL permit deterministic cleanup using:

* explicit release
* scoped binding handles
* context-manager or equivalent structured lifecycle support

Garbage collection alone SHALL NOT be the only cleanup strategy for active scope bindings.

* be timezone-aware
* use UTC
* remain immutable

29. Metadata Architecture

Metadata SHALL support immutable values composed of approved scalar and structured types.

Metadata SHALL NOT contain:

* open file handles
* network clients
* service instances
* mutable platform services
* credentials
* arbitrary executable objects

The exact serializable type policy SHALL be defined by the implementation specification within these architectural limits.

Metadata copying SHALL prevent caller mutation from altering an existing context.

⸻

30. Propagation Architecture

30.1 Explicit Propagation

Contexts SHALL be passable explicitly through public service operations where required.

30.2 Scoped Propagation

Within the same process, the service MAY provide execution-scope propagation through the approved scope store.

30.3 Asynchronous Tasks

A child asynchronous task MAY inherit the active context when supported by the runtime scope mechanism.

Changes to child-task bindings SHALL NOT modify the parent task’s binding.

30.4 Threads

Context SHALL NOT be assumed to propagate automatically to a newly created thread.

Thread propagation SHALL require explicit context passing and binding.

30.5 Processes and External Calls

Context SHALL NOT propagate automatically across:

* process boundaries
* command invocations
* network boundaries
* MCP connections
* AI-provider requests

Any external propagation format requires separate approved architecture.

32. Acceptance Criteria

The Platform Context Service architecture SHALL be considered implementation-ready only when the implementation specification can deterministically require all of the following.

32.1 Service Behavior

* The service initializes only after mandatory dependencies.
* The service creates immutable contexts.
* Every context has a service-generated unique context identifier.
* Correlation identifiers are generated or preserved correctly.
* Platform context is immutable and process-wide.
* Request contexts remain isolated.
* No implicit execution context is created by current().
* Required current-context lookup produces ContextNotFound when absent.
* Explicit binding establishes the current context.
* Nested bindings restore the prior context when enabled.
* Cloning creates a new context identity.
* Cloning preserves correlation identity by default.
* Cloning does not mutate the source.
* Release removes the active association.
* Shutdown clears service-owned active associations.
* Repeated release and shutdown operations are safely handled.

32.2 Dependency Compliance

* Configuration is obtained only through the Configuration Service.
* Logging is performed only through the Logging Service.
* Runtime discovery uses the Service Registry.
* Lifecycle state integrates with the Service Lifecycle Manager.
* Health integrates through the approved health-provider interface.
* No prohibited service dependency exists.

32.3 Data Integrity

* Contexts and platform metadata are immutable.
* Caller-owned mutable metadata cannot mutate existing contexts.
* Reserved keys cannot be overridden.
* Metadata limits are enforced.
* Secrets are not stored in context.
* Timestamps are UTC and timezone-aware.
* Released bindings cannot be reused.

32.4 Concurrency

* Independent threads do not share current contexts implicitly.
* Independent asynchronous tasks remain isolated.
* Concurrent context creation is safe.
* Concurrent lookup is safe.
* Concurrent release does not corrupt scope state.

32.5 Resource Management

* No unbounded registry of released contexts is retained.
* Explicit or structured release is supported.
* Partial initialization failure releases acquired resources.
* Shutdown is idempotent.
* Routine operations perform no network, repository, or filesystem I/O.

32.6 Errors and Diagnostics

* Canonical errors are exposed.
* Implementation-specific errors remain encapsulated.
* Required lifecycle and failure events are logged.
* Full context metadata is not logged.
* Health reports readiness without exposing context data.

32.7 Package Conformance

* The implementation resides under vibe.context.
* The public API is separated from internal components.
* Public exports are limited to approved interfaces, models, and errors.
* Tests cover public behavior and internal component contracts.

34. ADR Assessment

This addendum does not require an ADR because it:

* does not replace an approved architectural decision
* does not change the purpose of the Platform Context Service
* does not change the approved service ownership boundary
* does not introduce an external technology dependency
* does not change the approved implementation sequence
* does not alter another public service contract

An ADR SHALL be required if implementation proposes:

* persistent context storage
* automatic distributed context propagation
* cross-process context serialization
* Event Bus dependency
* Repository Service dependency
* external tracing infrastructure
* mutable execution contexts
* replacement of explicit lifecycle management with unmanaged global state
