# 1. Purpose

The purpose of this document is to define the functional and architectural boundaries of 86-vibe.

A clearly defined scope enables consistent product planning, prevents uncontrolled feature growth and provides an objective basis for evaluating future enhancement requests.

Every proposed capability should be assessed against this document before work begins.

---

# 2. Scope Statement

86-vibe is an AI Engineering Platform.

Its purpose is to coordinate the complete software engineering lifecycle by providing governance, documentation, orchestration, automation and reusable engineering assets.

86-vibe is **not** intended to replace existing development tools. Instead, it integrates with them to create a cohesive engineering operating model.

The platform focuses on **how software is engineered**, rather than on the business functionality of the software being built.

---

# 3. Product Boundaries

86-vibe operates above the implementation layer.

It coordinates engineering activities rather than replacing specialised tools.

The platform provides:

- engineering governance
- engineering knowledge
- AI orchestration
- automation
- project templates
- workflow management

It relies upon external systems for activities such as source control, issue tracking, code editing and deployment.

# 5. Out of Scope

The following capabilities are explicitly excluded from Version 1.0.

## 5.1 Source Code Hosting

86-vibe will not replace GitHub, GitLab or similar platforms.

---

## 5.2 IDE Functionality

86-vibe will not become an integrated development environment.

Cursor, Claude Code and other IDEs remain external tools.

---

## 5.3 AI Model Development

The platform will not train, fine-tune or host Large Language Models.

---

## 5.4 Cloud Hosting

86-vibe will not provide cloud infrastructure or application hosting.

---

## 5.5 CI/CD Platform

The platform will integrate with CI/CD solutions but will not replace them.

---

## 5.6 Identity Management

Enterprise authentication, user management and access control systems are outside the initial scope.

---

## 5.7 Billing and Licensing

Commercial licensing, subscriptions and payment systems are not included in Version 1.0.

---

## 5.8 Project-Specific Business Logic

Business functionality belongs within individual application repositories rather than within 86-vibe itself.

---

## 5.9 Monitoring Production Applications

Application monitoring and operational observability remain responsibilities of the deployed application.

86-vibe may integrate with monitoring platforms but does not replace them.

# 8. Guiding Scope Principles

When evaluating new capabilities, the following questions should be answered.

### Does this improve software engineering?

If not, it probably does not belong in 86-vibe.

---

### Does this duplicate another mature product?

If yes, integration should be preferred over replacement.

---

### Is this reusable across multiple projects?

If not, it should probably live within the application repository rather than the platform.

---

### Does this strengthen the engineering operating model?

If not, it should be reconsidered.

---

### Does this increase maintainability?

Features that increase complexity without improving maintainability should generally be rejected.

---

# 9. Scope Management

Changes to product scope shall follow the governance process.

The process is:

1. Proposal submitted.
2. Scope assessment completed.
3. Architectural impact reviewed.
4. ADR created if required.
5. Product Owner approval obtained.
6. Roadmap updated.
7. Jira backlog created.

No feature should be implemented before completing this process.

---

# 10. Summary

86-vibe is an engineering platform rather than an engineering tool.

Its responsibility is to coordinate, govern and improve software engineering rather than replace the specialised tools used during development.

The platform should remain focused on reusable engineering capabilities that benefit every project built upon it.

By maintaining disciplined product boundaries, 86-vibe will remain modular, maintainable and extensible while avoiding unnecessary duplication of existing technologies.

This scope statement establishes the boundaries against which all future feature requests and architectural decisions should be evaluated.
