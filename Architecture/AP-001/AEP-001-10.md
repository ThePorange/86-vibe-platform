# 1. Purpose

This document defines the versioning strategy for the 86-vibe platform.

The objective is to ensure that every significant engineering artefact can be uniquely identified, traced and managed throughout its lifecycle.

A consistent versioning strategy supports reproducibility, compatibility and controlled evolution of the platform.

---

# 2. Versioning Objectives

The versioning strategy SHALL:

- uniquely identify platform releases
- support traceability
- enable reproducible builds
- communicate compatibility
- minimise ambiguity
- support independent repository evolution
- simplify rollback when necessary

---

# 3. Versioning Principles

The following principles govern versioning within 86-vibe.

## VS-001 Consistency

A consistent versioning scheme SHALL be used across all repositories.

---

## VS-002 Traceability

Every released artefact SHALL have an identifiable version.

---

## VS-003 Independence

Repositories MAY evolve independently while maintaining compatibility declarations.

---

## VS-004 Transparency

Version changes SHOULD clearly communicate the significance of the change.

---

## VS-005 Immutability

Released versions SHALL be immutable.

Corrections SHALL be delivered through a subsequent version rather than modifying an existing release.

---

# 4. Semantic Versioning

86-vibe adopts Semantic Versioning (SemVer).

Version format:

```
MAJOR.MINOR.PATCH
```

### Major Version

Increment when introducing incompatible architectural or functional changes.

Example:

```
1.0.0 → 2.0.0
```

---

### Minor Version

Increment when introducing backwards-compatible functionality.

Example:

```
1.2.0 → 1.3.0
```

---

### Patch Version

Increment for backwards-compatible corrections.

Example:

```
1.3.4 → 1.3.5
```

---

Pre-release identifiers MAY be used.

Examples:

```
1.0.0-alpha.1

1.0.0-beta.2

1.0.0-rc.1
```

---

# 5. Repository Versioning

Each repository SHALL maintain an independent semantic version.

Repositories include:

- 86-vibe-platform
- 86-vibe-cli
- 86-vibe-example

Repository versions SHOULD evolve independently unless coordinated platform releases are required.

Repository releases SHALL identify compatible versions of dependent repositories.

Example:

| Repository | Version |
|------------|----------|
| 86-vibe-platform | 1.4.0 |
| 86-vibe-cli | 1.3.1 |
| 86-vibe-example | 1.4.0 |

---

# 6. Architecture Versioning

Architecture documents SHALL include document versions.

Major architectural changes SHALL be recorded using an Architecture Decision Record (ADR).

Minor clarifications MAY increment the minor version.

Editorial corrections SHOULD increment the patch version.

Example:

| Document | Version |
|----------|----------|
| AEP-001-07 | 1.0.0 |
| AEP-001-08 | 1.1.0 |
| AEP-001-09 | 1.0.2 |

Every document SHALL include a revision history.

---

# 7. Documentation Versioning

Documentation SHALL remain version controlled within Git.

Published documentation SHOULD identify:

- document version
- publication date
- revision history
- related architecture package

Documentation changes SHALL be reviewed using the standard pull request process.

---

# 8. Template Versioning

Reusable templates SHALL maintain independent versions.

Templates include:

- PRD Templates
- Architecture Templates
- ADR Templates
- Story Templates
- Review Templates

Template revisions SHOULD include migration guidance where practical.

---

# 9. Agent and Workflow Versioning

AI Agents SHALL maintain version identifiers.

Versioning applies to:

- agent definitions
- prompts
- workflow definitions
- routing policies
- reusable engineering assets

Changes affecting behaviour SHOULD increment the version.

Agent versions SHALL be documented within the platform repository.

---

# 10. Release Tags

Every repository release SHALL be identified using Git tags.

Recommended format:

```
v1.0.0
```

Architecture milestones MAY use additional tags.

Examples:

```
ap-001-approved

ap-002-approved

platform-v1.0.0
```

Release notes SHALL accompany every tagged release.

---

# 11. Compatibility

Repository releases SHOULD identify compatibility with dependent repositories.

Example:

| Component | Compatible Version |
|-----------|--------------------|
| Platform | 1.2.x |
| CLI | 1.2.x |
| Example | 1.2.x |

Where compatibility cannot be maintained, the change SHALL be documented through an ADR and release notes.

---

# 12. Summary

The Versioning Strategy establishes a consistent approach to identifying, managing and evolving engineering artefacts throughout the 86-vibe platform.

By adopting Semantic Versioning across repositories, architecture, documentation, templates and AI assets, the platform maintains traceability while allowing components to evolve independently.

All future releases SHALL comply with the versioning principles defined within this document.
