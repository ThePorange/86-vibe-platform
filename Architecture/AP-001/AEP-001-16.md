# 1. Purpose

This document defines the initial product backlog for the development of the 86-vibe platform.

The backlog provides the implementation roadmap for Version 1.0 and establishes the structure that SHALL be used for future work management.

Detailed implementation planning SHALL be managed within Jira.

---

# 2. Backlog Objectives

The backlog SHALL:

- organise engineering work
- support incremental delivery
- maintain traceability
- align with the Product Roadmap
- enable AI-assisted implementation
- support prioritisation
- preserve architectural integrity

---

# 3. Backlog Principles

The backlog SHALL follow these principles.

## BL-001

Every Epic SHALL deliver measurable business value.

---

## BL-002

Every Story SHALL belong to exactly one Epic.

---

## BL-003

Every Story SHALL have defined acceptance criteria.

---

## BL-004

Implementation SHALL begin only after Story approval.

---

## BL-005

Completed Stories SHALL update documentation where required.

---

## BL-006

Architecture changes SHALL reference approved ADRs.

---

# 4. Backlog Hierarchy

The work hierarchy for 86-vibe is:

```text
Vision
    │
    ▼
Roadmap
    │
    ▼
Epic
    │
    ▼
Story
    │
    ▼
Sub-task
```

The hierarchy SHALL remain consistent across all platform repositories.

---

# 5. Initial Product Backlog

The initial implementation backlog consists of the following Epics.

| Epic ID | Epic | Primary Outcome |
|---------|------|-----------------|
| EPIC-001 | Core Platform Foundation | Create the platform repository structure and engineering standards |
| EPIC-002 | Documentation Engine | Build documentation templates and generation capabilities |
| EPIC-003 | AI Agent Framework | Implement specialised engineering agents |
| EPIC-004 | Knowledge Management | Deliver reusable engineering knowledge capabilities |
| EPIC-005 | MCP Integration | Implement external tool integration |
| EPIC-006 | OpenRouter Intelligence | Implement AI provider abstraction and routing |
| EPIC-007 | Jira Automation | Automate engineering planning workflows |
| EPIC-008 | Bootstrap Engine | Generate new engineering projects |
| EPIC-009 | Reference Application | Build a complete demonstration project |

These Epics align directly with the Product Roadmap defined in AEP-001-12.

# 7. Story Standards

Every Story SHALL include:

- Story identifier
- User story
- Business value
- Acceptance criteria
- Dependencies
- Architecture references
- Related ADRs (where applicable)
- Test requirements
- Documentation requirements

Stories SHOULD remain independently implementable wherever practical.

---

# 8. Backlog Governance

The backlog SHALL be managed according to the Governance Model.

The Product Owner SHALL:

- prioritise Epics
- approve Stories
- accept completed work
- approve releases

AI MAY:

- propose Epics
- propose Stories
- refine acceptance criteria
- identify dependencies

AI SHALL NOT approve backlog items.

Changes to Epic scope SHALL follow the established governance process.

---

# 9. Summary

The Initial Jira Backlog establishes the implementation structure for the 86-vibe platform.

By organising work into strategic Epics aligned with the Product Roadmap, the backlog provides a clear transition from architecture into implementation.

Detailed Story creation, refinement and execution SHALL occur within Jira while maintaining full traceability to the architecture and governance established in AP-001.
