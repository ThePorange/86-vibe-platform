# 1. Purpose

This document defines the release strategy for the 86-vibe platform.

The objective is to ensure that every release is predictable, repeatable, traceable and governed by a consistent engineering process.

The release strategy applies to all repositories that form part of the 86-vibe platform.

---

# 2. Release Objectives

The release strategy SHALL:

- provide predictable software releases
- minimise release risk
- ensure documentation remains synchronised with implementation
- maintain compatibility across repositories
- support rollback where required
- preserve complete release traceability

---

# 3. Release Principles

The following principles govern every release.

## RS-001 Repeatability

Every release SHALL be reproducible from version-controlled artefacts.

---

## RS-002 Traceability

Every release SHALL be traceable to:

- Architecture Package(s)
- ADRs
- Jira Epics
- Jira Stories
- Git commits
- Pull Requests

---

## RS-003 Quality

Only work satisfying defined acceptance criteria SHALL be released.

---

## RS-004 Documentation

Documentation SHALL be updated before release approval.

---

## RS-005 Automation

Release activities SHOULD be automated wherever practical.

Automation SHALL complement, not replace, governance.

---

# 4. Release Types

86-vibe supports the following release types.

## Alpha

Purpose

Early internal validation.

Characteristics

- incomplete functionality
- frequent changes
- limited stability

---

## Beta

Purpose

Feature-complete evaluation.

Characteristics

- broader testing
- limited production readiness
- feedback collection

---

## Release Candidate (RC)

Purpose

Final validation before production release.

Characteristics

- release complete
- defect correction only
- production verification

---

## General Availability (GA)

Purpose

Production release.

Characteristics

- fully approved
- documented
- supported
- version tagged

---

## Patch Release

Purpose

Correct defects without introducing new functionality.

Characteristics

- backward compatible
- limited scope
- targeted testing

---

# 5. Release Lifecycle

Every release SHALL follow the lifecycle below.

```text
Planning
    │
    ▼
Implementation
    │
    ▼
Testing
    │
    ▼
Documentation
    │
    ▼
Release Review
    │
    ▼
Approval
    │
    ▼
Tag Release
    │
    ▼
Publish
```

Each stage SHALL complete successfully before progressing to the next.

---

# 6. Release Readiness

A release SHALL NOT proceed until the following criteria have been satisfied.

## Architecture

- Approved architecture
- Approved ADRs

---

## Development

- All planned work completed
- Code reviewed
- No unresolved critical defects

---

## Testing

- Unit tests passed
- Integration tests passed
- Regression tests passed
- Acceptance criteria satisfied

---

## Documentation

- User documentation updated
- Technical documentation updated
- Architecture updated where required
- Changelog completed
- Release notes prepared

---

## Versioning

- Repository version updated
- Release tag prepared
- Compatibility verified

---

# 7. Release Deliverables

Every release SHALL include:

- Source code
- Version tag
- Release notes
- Changelog
- Updated documentation
- Compatibility information
- Known issues (if applicable)

Repositories SHOULD also include installation or upgrade guidance where appropriate.

---

# 8. Release Approval

Release approval authority is defined below.

| Release Stage | Approval Authority |
|---------------|-------------------|
| Alpha | Product Owner |
| Beta | Product Owner |
| Release Candidate | Product Owner |
| General Availability | Product Owner |
| Patch Release | Product Owner |

Future enterprise deployments MAY introduce additional approval authorities.

No AI agent SHALL independently approve a release.

---

# 9. Post-Release Activities

Following every release the following activities SHOULD be completed.

- Verify published artefacts.
- Confirm repository tags.
- Archive release documentation.
- Update project roadmap.
- Review release metrics.
- Record lessons learned.
- Create backlog items for follow-up improvements where appropriate.

The objective is to ensure that every release contributes to the continuous improvement of both the platform and its engineering processes.

---

# 10. Summary

The Release Strategy establishes a consistent and governed approach to delivering software within the 86-vibe platform.

By combining structured approval gates, comprehensive documentation, version control and repeatable workflows, the platform ensures that releases remain predictable, traceable and maintainable.

All future releases SHALL comply with the strategy defined within this document unless an approved Architecture Decision Record explicitly authorises an alternative approach.
