# 1. Purpose

This document defines the technology strategy for 86-vibe.

The objective is to establish the principles that govern technology selection while identifying the initial technology stack for Version 1.0.

Technology choices SHALL support the product vision, architectural principles and long-term maintainability of the platform.

---

# 2. Technology Objectives

The technology strategy has the following objectives.

- Support long-term maintainability.
- Minimise vendor lock-in.
- Encourage modular architecture.
- Enable AI-assisted engineering.
- Promote open standards where practical.
- Support automation throughout the software development lifecycle.
- Allow technologies to evolve without major architectural redesign.

---

# 3. Technology Principles

## TS-001 Vendor Neutrality

Technologies SHOULD be selected to minimise dependence upon a single vendor.

Where abstraction layers are available, they SHOULD be preferred.

---

## TS-002 Open Standards

Open standards SHOULD be preferred over proprietary alternatives where they provide comparable capability.

---

## TS-003 Replaceability

Every major technology SHOULD be replaceable with minimal architectural impact.

---

## TS-004 Simplicity

Technologies SHOULD be selected based upon their suitability rather than popularity.

---

## TS-005 Community Support

Preference SHOULD be given to technologies with active communities, good documentation and long-term support.

---

## TS-006 Automation

Selected technologies SHOULD support automation through APIs, command-line interfaces or standard protocols.

---

## TS-007 Interoperability

Technologies SHOULD integrate through well-defined interfaces rather than proprietary coupling.

---

# 5. AI Technology Strategy

86-vibe SHALL remain independent of individual AI providers.

OpenRouter is adopted as the primary AI gateway because it provides access to multiple model providers through a common interface.

AI models SHALL be selected according to the nature of the engineering task rather than using a single model for all activities.

Typical routing categories include:

- Strategic reasoning
- Architecture
- Planning
- Software implementation
- Documentation
- Testing
- Review

The detailed routing policy will be defined in a later architecture package.

---

# 6. Integration Strategy

86-vibe integrates with external systems rather than replacing them.

Integration SHALL use published interfaces wherever practical.

Primary integration mechanisms include:

- MCP
- REST APIs
- Git repositories
- Command-line interfaces
- Standard file formats

Integrations SHALL remain modular to simplify replacement or extension.

---

# 7. Documentation Strategy

Documentation is a core engineering artefact.

The platform adopts the following documentation standards.

- Markdown SHALL be the primary authoring format.
- Mermaid SHALL be used for architecture diagrams.
- Documentation SHALL be maintained within Git repositories.
- Documentation SHALL evolve alongside implementation.
- Architecture documentation SHALL remain within the platform repository.

Future documentation publishing tools MAY be adopted without changing the underlying Markdown source.

---

# 8. Development Strategy

Development follows a Git-based workflow.

Engineering activities SHOULD be automated wherever practical.

All code SHALL:

- be version controlled
- be reviewed
- include appropriate tests
- satisfy acceptance criteria
- update relevant documentation

The preferred development approach is incremental delivery through small, independently reviewable changes.

---

# 9. Evaluation Criteria

New technologies SHALL be evaluated against the following criteria.

| Criterion | Description |
|-----------|-------------|
| Architectural Fit | Supports platform architecture |
| Maintainability | Long-term sustainability |
| Community | Active ecosystem |
| Documentation | High-quality documentation |
| Automation | Supports automation |
| Security | Acceptable security posture |
| Integration | Works with existing platform |
| Cost | Sustainable operational cost |
| Replaceability | Can be substituted if required |

No technology SHALL be adopted solely because it is fashionable or recently released.

---

# 10. Technology Lifecycle

Technologies progress through the following lifecycle.

## Evaluation

Technology is being investigated.

---

## Approved

Technology has been accepted for platform use.

---

## Preferred

Technology is recommended for new development.

---

## Deprecated

Technology remains supported but SHOULD NOT be selected for new work.

---

## Retired

Technology is no longer supported by the platform.

Changes in lifecycle status SHALL be documented through an Architecture Decision Record.

---

# 11. Summary

The Technology Strategy establishes the principles used to select, adopt and evolve technologies within 86-vibe.

Rather than binding the platform to individual vendors or products, this strategy promotes modularity, interoperability and long-term maintainability.

Technology selections SHALL support the architectural principles defined in previous documents while remaining flexible enough to accommodate future innovation.

The technology strategy defined within this document SHALL guide all future technology decisions for the platform.
