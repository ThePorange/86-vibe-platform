# 1. Purpose

This document defines the governance framework for 86-vibe.

Governance establishes how engineering decisions are made, approved, recorded and reviewed throughout the software development lifecycle.

The objective is to balance AI-driven automation with appropriate human oversight while maintaining consistency, quality and accountability.

---

# 2. Governance Objectives

The governance model exists to ensure that:

- engineering decisions remain traceable
- software quality remains consistent
- architecture evolves in a controlled manner
- AI operates within defined boundaries
- engineering standards are applied consistently
- documentation remains accurate
- releases are predictable and auditable

---

# 3. Governance Principles

The governance model SHALL follow the Design Principles defined in AEP-001-04.

Additionally:

**GOV-001** – Governance SHALL enable engineering rather than obstruct it.

**GOV-002** – Approval SHALL occur at defined lifecycle stages.

**GOV-003** – Every significant decision SHALL have an accountable owner.

**GOV-004** – Architectural changes SHALL be documented.

**GOV-005** – Automation SHALL not remove accountability.

---

# 4. Decision Authority

Decision-making authority is assigned as follows.

| Decision | Authority |
|-----------|-----------|
| Product Vision | Product Owner |
| Product Roadmap | Product Owner |
| Product Scope | Product Owner |
| Architecture | Chief Systems Architect |
| ADR Approval | Product Owner + Chief Systems Architect |
| Technology Selection | Chief Systems Architect |
| Story Prioritisation | Product Owner |
| Implementation | AI Delivery Agents |
| Code Review | Human Reviewer |
| Release Approval | Product Owner |

No AI agent SHALL approve its own work.

---

# 5. Governance Structure

The governance hierarchy is:

```text
Product Vision
        │
        ▼
Architecture
        │
        ▼
Engineering Standards
        │
        ▼
Workflows
        │
        ▼
Implementation
```

Each layer SHALL inherit the rules of the layer above it.

Implementation SHALL NOT redefine architectural principles.

# 7. Architecture Governance

Architecture SHALL evolve through Architecture Decision Records (ADRs).

Architectural changes include:

- technology changes
- repository restructuring
- workflow changes
- governance changes
- security changes
- AI model strategy
- integration strategy

No architectural change SHALL be implemented without an approved ADR.

---

# 8. Documentation Governance

Documentation SHALL be maintained alongside implementation.

Engineering work SHALL NOT be considered complete until required documentation has been updated.

Documentation reviews SHOULD occur as part of every pull request.

Generated documentation SHALL be reviewed before publication.

---

# 9. AI Governance

AI operates as an engineering participant rather than an autonomous decision maker.

AI SHALL:

- implement approved work
- generate documentation
- propose improvements
- execute approved workflows

AI SHALL NOT:

- redefine product scope
- approve releases
- modify governance
- bypass approval gates
- introduce architectural changes without review

---

# 10. Change Management

Changes SHALL follow a controlled process.

1. Proposal
2. Impact Assessment
3. Architecture Review
4. ADR (if required)
5. Product Owner Approval
6. Jira Planning
7. Implementation
8. Testing
9. Documentation Update
10. Release

Emergency changes SHALL be documented retrospectively.

---

# 11. Exception Management

Exceptions SHOULD remain rare.

An exception requires:

- documented justification
- identified risks
- approval authority
- review date

Temporary exceptions SHALL include an expiration date.

Expired exceptions SHALL be reviewed before renewal.

---

# 12. Compliance

Compliance is measured through periodic governance reviews.

The following SHOULD be reviewed:

- documentation completeness
- architecture compliance
- coding standards
- test coverage
- AI usage
- ADR compliance
- release quality

Compliance metrics SHALL be incorporated into future engineering dashboards.

---

# 13. Summary

The governance model provides the operational framework that enables humans and AI to collaborate effectively while preserving engineering discipline.

Governance is intended to support engineering quality, maintain accountability and ensure that the platform evolves in a controlled and transparent manner.

All future engineering activities within 86-vibe SHALL comply with this governance model unless an approved ADR explicitly authorises an exception.
